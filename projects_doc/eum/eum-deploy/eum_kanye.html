Runtime Architecture: What EUM Is Operating

EUM is a multi-tenant, on-demand telemetry aggregation system. It receives browser/mobile/IoT/synthetic beacons, partitions by account and application GUID, aggregates per minute, and exposes metrics and snapshots for Controllers.

Beacon to Controller: canonical metric path

Browser / Mobile / IoTAgents send JSON beacons over HTTPS/HTTP. Authentication is usually not on the ingest edge; validation happens downstream.
Collector proxy + ALBReceives global traffic and forwards to the right collector zone.
Collector podsValidate, enrich with geo/user-agent, classify slow/stall, sample snapshots, and create L1 aggregates.
L1 storeHistorically S3 Level One; in the DFD, Redis holds one-minute account-level aggregates from each collector pod.
Aggregator + L2One node wins processing rights per account, merges collector outputs, compresses, and writes final L2 metrics for Controllers.
Account/App GUIDMulti-tenancy is built around globally unique account and application identifiers.
DynamoDB / CosmosStores account metadata, application keys, configuration, and optimistic concurrency state.
S3 / BlobStores metrics, events, snapshots, source maps, crash reports, and synthetic artifacts with retention windows.
Controller downloadControllers request the previous minute after applying EUM cloud time skew.

Collector responsibilities

  • Receive and validate browser/mobile/IoT/synthetic telemetry.
  • Resolve geo, browser, device, OS, page, AJAX, and classification dimensions.
  • Aggregate in memory by application within the one-minute cycle.
  • Write unique per-collector L1 tuples so no collector overwrites another collector.
  • Keep Jetty request threads decoupled from heavier processing via queues and worker pools.

Aggregator responsibilities

  • Wake after the minute boundary and merge all collector tuples for an account/application.
  • Use DynamoDB conditional updates to ensure one aggregator processes one account per minute.
  • Write final L2 metric objects and sampled events/snapshots.
  • Expose Controller APIs for metric download and account/application management.
  • Remain stateless so any pod can process any account in the next cycle.

Collector internals: async three-stage processing

Jetty request threadsReceive beacons and quickly validate request shape so front-door threads return to accepting traffic.
Concurrent queueDecouples network IO from enrichment and aggregation work.
Dimension workersAdd geo, browser, OS, device, page/AJAX, and slow/stall classification dimensions.
In-memory app mapMerge metrics by application GUID and metric key with synchronization around map creation and value merge.
Parallel uploaderAt minute boundary, swap the active map and upload compressed L1 tuples in parallel.

Aggregator internals: ownership, merge, publish

Timer wakes T+15sAggregator waits for collectors to finish the previous minute's L1 writes.
Read account rowsMetadata rows in DynamoDB/Cosmos identify accounts and applications.
Conditional update winsOnly one node updates last_updated_time for an account and owns that batch.
Read all L1 tuplesQuery by timestamp and application/account prefix, then merge all collector outputs.
Write L2 and snapshotsCompress final metrics/events and expose them for Controller/API reads.
T+00s
Collectors close the previous minute. They swap in-memory stores, continue accepting new traffic, and upload one-minute L1 batches with timestamp rounded to the minute start.
T+15s
Aggregators start processing. Each aggregator competes for account rows using conditional updates. Winners read all L1 tuples for that account/application and merge them.
T+25s
Aggregation should be complete. Final L2 objects are compressed and available for downstream reads.
T+40s
Controllers download. Controllers ask for the previous minute after applying EUM cloud time skew. Clock discipline is part of data integrity.
Interview signal: the stateless claim is not hand-wavy. Statelessness works because collector writes are unique, aggregator ownership is protected by conditional metadata updates, and durable state lives in Redis/S3/DynamoDB or their Azure equivalents.

Kanye Orchestration Stack

Kanye is the delivery and operations control plane. It is not the service source code; it packages, deploys, scales, upgrades, and recovers the platform.

Complete delivery stack

CI trigger layerTeamCity Kotlin DSL plus legacy Jenkins jobs. Parameters: ENV, REGION, TIER, BRANCH, BUILD, COMMAND.
Wrapper layerHundreds of tiny directories such as eum-prod-deploy, sum-shadow-autoscale, eum-sydney-drain-node.
Real implementationsAbout ten scripts do the real work: deploy, scale, drain, update EKS, backup DynamoDB, batch instances, delete pods, kill KPL.
AWS + KubernetesEKS clusters, Helm releases, HPAs, ASGs, ALBs, DynamoDB, Redis, S3, Kinesis, SQS, and Istio.

Core insight 1

Operations are the same script with different parameters. Encode the parameters in directory names and wrapper environment variables.

Core insight 2

A microservice deploy is helm upgrade --install with the correct values cascade, image tag, secrets file, and kube context.

Core insight 3

Stateful operations need explicit, idempotent tooling because Kubernetes primitives alone do not solve migrations, backups, KPL hangs, or node drains.

Helm Deployment Mechanics

One deploy script serves all services. Its job is to validate inputs, switch context, assemble values, pin the image, run Helm, and wait for rollout.

Values cascade

1. Environmentenvironment.yamlglobal defaults
2. Tierrole.collector.yamlHPA, probes, limits
3. Regionrole.collector-us-west-2.yamlregional overrides
4. Secretssecrets.prod.yamllast wins
5. Image--set image.tag=master-1234explicit pin

The command shape

helm secrets upgrade -i \
  -f configs/prod/environment.yaml \
  -f configs/prod/role.aggregator.yaml \
  -f configs/prod/role.aggregator-us-west-2.yaml \
  -f configs/prod/secrets.prod.yaml \
  --set image.tag=master-1234 \
  --reset-values \
  eum-aggregator ./charts/aggregator

--reset-values matters: every release starts from the files and explicit flags. Old ad-hoc --set values cannot silently survive.

Umbrella chart

docker/helm/eum owns the parent chart and service subcharts. Roughly 50 service charts share one common template library.

Common templates

Deployment, HPA, PDB, Service, ConfigMap, ServiceAccount, probes, lifecycle hooks, and annotations are centralized.

Checksum rollout

ConfigMap/Secret checksums become pod annotations. A config change mutates the pod template, creating a new ReplicaSet automatically.

Schema chart exception: migration jobs are run-to-completion resources, so the release is deleted and recreated before rerun. Long-running services can upgrade in place; one-shot jobs often should not.

Multi-Region Configuration

Region awareness is stored as configuration, not scattered through deployment code. The scripts stay region-agnostic; the JSON and values files carry the region facts.

What region config contains

  • Primary region and region list.
  • Subnets mapped to AZs.
  • AMI IDs per region.
  • Security groups and utility client groups.
  • Per-tier ASG desired, min, max, cooldown, health grace period, and instance types.
  • Drain timing such as drain_sec.

Region switch flow

export ENV=prod
export REGION=eu-west-1
export TIER=collector

kubectl config use-context eu-west-1
cd ../docker/helm/eum
./deploy.sh

The entire targeting mechanism is environment variables plus the active kubeconfig context.

Why prod-deploy1 through prod-deploy6? Parallel CI jobs get separate workspaces, independent audit histories, and independent failures. One slow or failed region does not serialize the world.

Two indexing axes

The config model is indexed by region and tier. Region answers where to run; tier answers what kind of nodes and limits a service needs.

SSL endpoint maps

Environment and region mapping also captures public endpoint differences so jobs do not bake endpoint logic into scripts.

Config over code

Adding or changing region capacity should usually mean editing JSON/YAML values, not branching Python or Bash logic.

Scaling Architecture

EUM scales in layers: pods first, nodes second, operators for predictable events.

Three-layer scale response

Layer 1: HPARuns every few seconds. CPU target around 50 percent gives JVM services headroom before saturation.
Layer 2: Cluster AutoscalerWatches Pending pods, simulates whether a new node helps, then adjusts ASG desired count.
Layer 3: ASGLaunches EC2 nodes, waits for kubelet join, and uses lifecycle hooks for graceful termination.
Manual overrideUsed for known events, Black Friday style spikes, or incident response when humans have better context.

HPA math

desired = ceil(currentReplicas * currentCPU / targetCPU)

currentReplicas = 3
currentCPU = 80
targetCPU = 50

desired = ceil(3 * 80 / 50) = 5

Why 50 percent target?

JVM warmup and readiness delay mean new capacity is not instant. Scaling at 50 percent leaves room for two minutes of pod startup and JIT/GC stabilization.

ASG cooldowns around 780 seconds prevent overreacting to short spikes after nodes are launched.

Custom rolling restart script

rolling_restart.sh exists for restart-only operations where the image does not change. The safe pattern is to select pods for a tier, delete or restart one disruption window at a time, wait for replacements to become Ready, and let PDBs block unsafe evictions. It is the same availability principle as a deploy, but without producing a new Helm revision.

PDB closes the gap: drains and other voluntary disruptions are blocked if eviction would violate minimum availability. RollingUpdate protects deployment; PDB protects operations.

Node Lifecycle: Drain, Cordon, Recycle

Node operations are idempotent because CI runners can die, AWS APIs can partially succeed, and humans need safe reruns at 3am.

Idempotent drain state machine

1. Check tagsIf EC2 tag processStatus=Detached exists, skip detach.
2. Detach ASGRemove instance from ASG. Replacement node starts before old pods are evicted.
3. CordonMark node unschedulable and label it processStatus=Cordoned.
4. DrainEvict non-daemonset pods, respecting PDBs and allowing ephemeral local data deletion.
5. TerminateTerminate EC2 and delete the Kubernetes node object after it is empty.
CommandMeaningWhen to use
kubectl cordonStops new pods from landing on a node. Existing pods keep running.Suspect a node, prepare for maintenance, or pause scheduling.
kubectl drainCordons then evicts pods, waiting for PDB-safe replacement.Node replacement, AMI rotation, K8s upgrade, ASG lifecycle termination.
ASG detachSeparates old instance from capacity accounting so replacement capacity can join first.Before draining to avoid capacity cliff.

EKS Control-Plane Upgrades

The playbook separates control plane, core add-ons, network add-ons, and node rotation so each phase has a small blast radius.

1
Validate skew. Upgrade only one minor version at a time. Node kubelet versions must remain within supported skew.
2
Upgrade control plane. aws eks update-cluster-version, then poll until successful. Nodes are still old, which is allowed temporarily.
3
Patch kube-proxy and CoreDNS. Roll DaemonSet/Deployment images to versions compatible with the target Kubernetes version.
4
Apply VPC CNI. Use WARM_IP_TARGET=2 for faster pod startup and remove conflicting WARM_ENI_TARGET.
5
Rotate nodes separately. Update ASG launch configuration or AMI, then drain old nodes in AZ-aware batches.
Why decouple node rotation? Control-plane changes can complete while workload capacity remains steady. Node replacement can then be done gradually, one balanced batch at a time.

AZ-Aware Scheduling and Drain Batching

EUM aims to spread replicas across AZs, but prefers serving traffic over perfect balance when capacity is tight.

Soft anti-affinity

preferredDuringSchedulingIgnoredDuringExecution tells the scheduler to spread pods across zones when possible. If one AZ is full, pods can still schedule elsewhere.

For stateless compute, soft spread avoids self-inflicted outages from overly strict placement rules.

Modern alternative

topologySpreadConstraints can express max skew directly. A modern implementation could use ScheduleAnyway for soft behavior or DoNotSchedule for hard behavior.

Placement inspection

getZonePerService_v2.py is the audit tool for the scheduler's actual result. It answers the operational question that YAML alone cannot: are pods really distributed across AZs right now, or did capacity pressure force an imbalance?

Round-robin node batches

Raw ASG orderAZ-a: a1 a2 a3
AZ-b: b1 b2 b3
AZ-c: c1 c2 c3
Batch 1a1, b1, c1: one node per AZ.
Batch 2a2, b2, c2: capacity remains balanced.
Batch 3a3, b3, c3: no AZ is emptied first.

Istio Service Mesh

In this architecture, Istio is used primarily for transparent service-to-service mTLS, not as the main deployment or ingress control plane.

What it adds

Envoy sidecars intercept traffic, present SPIFFE/X.509 identity, encrypt east-west traffic, and authenticate peers without application code changes.

Why permissive mode

Permissive mode accepts both mTLS and plaintext, allowing incremental migration across roughly 60 services.

What is not used

No heavy reliance on VirtualService canaries, mesh ingress, rate limiting, or circuit breaking. ALBs and RollingUpdate remain the main operational path.

Maturity point: the strongest interview answer is not "use every Istio feature." It is "use the feature we can operate safely, prove coverage, then tighten to STRICT when ready."

Schema Migrations and Data Safety

Schema and metadata changes are treated as stateful operations with backups and phased compatibility.

Two-phase schema pattern

Backup firstSnapshot DynamoDB tables before risky deploys and garbage collect old backups after retention.
Before-code migrationAdd new columns/indexes/attributes while keeping old readers and writers valid.
Deploy app codeRollingUpdate may run old and new pods at the same time, so both schema shapes must work.
After-code cleanupDrop unused fields or old paths only after no running code depends on them.

The reason is rollback reality: code rollback is cheap; destructive schema rollback is not. Compatibility windows let mixed-version pods coexist during rollout.

Secrets Management

Secrets are encrypted in git and decrypted only for deployment through SOPS and helm-secrets.

Why SOPS + helm-secrets

  • Git contains ciphertext, not plaintext credentials.
  • AWS KMS controls who can decrypt.
  • Helm receives decrypted values during upgrade.
  • Kubernetes stores rendered Secret objects in etcd, relying on cluster encryption at rest.

Why secrets file is last

Helm values are last-wins. If secrets.prod.yaml is last, lower-priority role or region files cannot accidentally override production credentials with defaults.

CI/CD and Versioning

TeamCity answers when and in what order; Make answers how each build/package/publish operation works.

TeamCity DSL

Kotlin build definitions hold VCS triggers, build steps, branch filters, and orchestration sequencing.

Make targets

Helm linting, Docker/ECR login, chart packaging, chart publish, and build number creation are local-runnable targets.

Calendar versions

Versions such as 26.5.1234 communicate build time without debating semantic major/minor meaning for high-frequency chart releases.

Migration value: if TeamCity is replaced later, the Makefile behavior can stay. CI should invoke delivery logic, not become delivery logic.

Disaster Toolbox

The operational philosophy is simple: operators run named jobs, not remembered shell commands.

KPL hang recovery

Kinesis Producer Library native process can deadlock. The runbook kills the native binary and relies on the Java supervisor to respawn a healthy producer.

Delete pod

Emergency pod deletion forces Deployment to create a replacement while PDBs prevent reducing the service below safe capacity.

k8s-recycler

Validates cluster health, respects PDBs globally, and rotates nodes with bounded disruption instead of a blind per-node loop.

Interview Master Q&A

These are the high-signal answers to practice until they feel natural.

Q1. Walk me through a production deploy end to end.
Engineer merges to a release branch. TeamCity publishes the image and chart through Make targets. Operator triggers the right eum-prod-deployN job with region, tier, and build. The wrapper sets env vars and calls deploy.sh. The deploy script switches kube context, builds the values cascade, pins image.tag, runs helm secrets upgrade -i --reset-values, and waits for Kubernetes rollout. RollingUpdate uses maxUnavailable=0, maxSurge=1, readiness, minReadySeconds=120, PDBs, and preStop drains.
Q2. How does EUM handle a sudden 5x traffic spike?
CPU rises on collector pods. HPA computes desired replicas from current CPU versus the 50 percent target. If existing nodes have capacity, new pods schedule and warm up. If pods become Pending, Cluster Autoscaler increases ASG desired capacity. ASG launches nodes, kubelet joins them, and the scheduler places pods. When the spike ends, HPA scales down pods and Cluster Autoscaler later drains and removes underutilized nodes through lifecycle hooks.
Q3. How do you upgrade Kubernetes without downtime?
Validate one-minor-version upgrade rules. Upgrade the EKS control plane first. Patch kube-proxy, CoreDNS, and VPC CNI. Use WARM_IP_TARGET to keep pod IP assignment fast. Rotate nodes as a separate operation by updating ASG launch config or AMI, then draining old nodes in AZ-aware batches. PDBs and replacement-first drain keep services available.
Q4. A pod is stuck in CrashLoopBackOff. What do you do?
Start with kubectl describe pod for events, not deletion. Check kubectl logs --previous for the last crash. Distinguish OOMKilled, image pull failures, liveness probe failures, and application exit errors. Confirm resource limits and recent image tag. Use the delete-pod job only as mitigation; root cause is code or config followed by redeploy.
Q5. How do you prevent a bad deploy from taking production down?
Use helm diff/deployment-plan, DynamoDB backup before schema risk, RollingUpdate with zero unavailable capacity, readiness probes that stop bad pods from receiving traffic, minReadySeconds for stabilization, PDBs during voluntary disruptions, and Helm rollback or emergency kubectl set image if needed. The best safety mechanism is that old pods keep serving if new pods cannot prove readiness.
Q6. Why keep all state external to EKS?
Nodes and clusters become replaceable compute. Draining a node is safe because state is in Redis, DynamoDB/Cosmos, S3/Blob, Kinesis/Kafka, SQS, or Elasticsearch. Cluster replacement becomes deploy services and shift traffic; no PVC migration is required. The trade-off is stronger dependency on managed data services, which is acceptable because the operational simplicity is enormous.
Q7. What would you improve?
Move wrappers toward matrix CI or GitOps where access/audit needs allow it. Add typed config validation for region/tier settings. Consider topology spread constraints over older anti-affinity. Evaluate progressive delivery through Istio only after the team is ready to operate VirtualServices. Consider External Secrets Operator for runtime rotation, balanced against the simplicity of SOPS.
Q8. How do you safely drain a node?
Detach the instance from the ASG so replacement capacity starts first, tag the EC2 instance as detached, wait for new nodes, cordon the Kubernetes node, label it cordoned, drain with daemonsets ignored and PDBs respected, then terminate EC2 and delete the node object. The v2 script is idempotent because tags and labels allow a failed run to resume rather than duplicate work.
Q9. How do you do schema migrations without downtime?
Run a backup first. Then apply a backward-compatible before-code migration that adds new schema while old code still works. Deploy application pods with RollingUpdate while old and new versions coexist. Finally run after-code cleanup only when no running pod depends on the old schema. Schema charts are treated as Kubernetes Jobs, so the release is purged and recreated for a fresh run.
Q10. Why one wrapper per environment and operation?
It matches legacy Jenkins/TeamCity job discovery, gives each job independent audit history and access control, lets prod regions run in parallel with separate workspaces, and is easy to grep. The downside is directory count, but the wrappers are intentionally thin; the real implementation surface stays small.
Q11. How are secrets managed?
Secret values files are encrypted with SOPS and protected by KMS. helm secrets decrypts them during deploy and passes them to Helm. The secrets file is placed last in the values cascade so lower-priority environment, role, or region files cannot override production credentials.
Q12. How do you debug pods not spreading across AZs?
First confirm the chart affinity or topology configuration. Then inspect actual placement with getZonePerService_v2.py or equivalent kubectl get pod -o wide checks. If placement is imbalanced, check node capacity per AZ, taints, labels, pending pods, and whether soft anti-affinity was intentionally relaxed because one AZ lacked capacity.
Q13. What does the disaster recovery toolbox include?
KPL kill jobs for hung native producers, delete-pod for emergency replacement, drain-node for bad nodes, kube-status and node-status for read-only triage, manage-account and local-admin for tenant operations, deployment-plan and diff for pre-flight review, DynamoDB backups for metadata risk, and k8s-recycler for cluster-wide AMI or node rotation.
Q14. How do chart and image versions work?
CI uses calendar-style chart versions like 26.5.1234 so operators can infer when an artifact was built. Images are pinned explicitly, often as branch plus build number, rather than relying on latest. Chart version and image version are decoupled: a chart can change without a new app image, and an app image can deploy through the same chart templates.

Closing Narrative

EUM at scale is the runtime architecture plus the operational discipline around it. Browser and mobile beacons flow into stateless collectors, one-minute L1 aggregation, aggregator ownership through DynamoDB conditional updates, L2 metric storage, and Controller downloads. Kanye keeps that machinery alive across regions by turning deployment and incident response into repeatable jobs: Helm for declarative service delivery, layered values to avoid drift, HPA plus ASG for elasticity, PDBs and preStop hooks for safe disruption, AZ-aware drains for node replacement, and coded runbooks for the ugly moments.

One sentence to remember: state is external, nodes are cattle, deployments are declarative, and operators run jobs instead of improvising commands.
