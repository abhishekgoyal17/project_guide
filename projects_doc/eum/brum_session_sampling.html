#2. Beacon Data — Where Does It Go?

#A browser beacon is an HTTP POST with a binary or JSON payload. Here is its lifecycle:

Step 1: Browser JS agent sends HTTP POST beacon
        Payload example:
        {
          "sessionId":   "sess_abc123",
          "appKey":      "EX-AAA-AUM",
          "startTime":   1698765432100,
          "endTime":     1698765445200,
          "pageUrl":     "https://example.com/checkout",
          "ajaxCalls":   [...],
          "jsErrors":    [...],
          "resourceTimings": [...]
        }

Step 2: HTTP Servlet in Collector receives it
        → Calls BeaconProcessor.processBeacon(rawBytes)

Step 3: BeaconProcessor parses raw bytes → BrowserRecord Java object
        BrowserRecord {
          sessionId:   "sess_abc123"
          appKey:      "EX-AAA-AUM"
          startTimeMs: 1698765432100
          endTimeMs:   1698765445200
          ...other fields
        }
        ↑ This object lives on JVM heap. It is NOT written to disk or DB.

Step 4: BrumSampler.sampleBrowserRecord(record) is called
        → Reads/writes ONLY to Redis (my project)
        → Returns true (accept) or false (drop)

Step 5a: If ACCEPTED → push BrowserRecord to Kinesis stream
         → downstream Sessionizer picks it up
         → eventually lands in EuM Custom View Store

Step 5b: If REJECTED → record is discarded from memory
         → reportUnsampledBeaconCount() called
         → nothing written anywhere (beacon is dropped)
Beacon data itself is never persisted by Collector. It is either forwarded downstream or dropped.

To provide a complete picture of the ingestion flow, we need to look at the Edge-of-the-Edge: the Content Delivery Network (CDN).

In the EUM Cloud architecture, the flow doesn't start at the ALB; it starts with the global distribution of the monitoring code itself.

#1. The Global Distribution: JS Agents on the CDN

Before a single metric is captured, the Browser Agent (JS agent) must be loaded into the user's browser.

Regional Deployment: The JS agent is hosted on CDNs (like CloudFront or Akamai) across multiple global regions. This ensures that when a user in London or Tokyo opens a monitored website, the agent script is fetched from a nearby "Edge Location" rather than a central server, minimizing page load impact.

Version Management: CDNs allow for "staged rollouts" of agent updates. Different regions can host different versions of the agent, allowing for canary testing of new collection features.

#2. Data Collection: The Beacon API

Once the agent is running in the user's browser, it hooks into the Beacon API and other browser performance interfaces.

The "Collection" Phase: The agent gathers page load timings, AJAX calls, and JS errors. It packages this into a binary or JSON payload known as a "Beacon."

The Transmission Phase: Instead of sending data back to the CDN (which is primarily for delivering content), the agent sends an HTTP POST request directly to the EUM Cloud Ingress.

#1. The Entry Point: AWS Application Load Balancers (ALB)

The ALB is the "External Facing" tier. Its primary job is Layer 7 (HTTP/HTTPS) termination and initial traffic segregation.

  • Traffic Segregation: By using different ALBs for different traffic types (BRUM vs. MRUM vs. ThousandEye), the system achieves fault isolation. If the Collector-proxy ALB is overwhelmed by a spike in Browser traffic, the TE-Collector ALB remains unaffected, ensuring ThousandEye monitoring persists.
  • SSL/TLS Termination: The ALBs handle the compute-heavy process of decrypting HTTPS traffic from the public internet. This offloads CPU cycles from the internal pods.
  • Scalability: ALBs are managed services that scale automatically to handle throughput spikes. They act as the "shock absorbers" for the EUM cloud.

#2. The Intelligence Layer: Envoy Proxy Tiers

Once the ALB accepts a request, it passes it to Envoy, a high-performance C++ distributed proxy. While the ALB is great at basic load balancing, Envoy provides the fine-grained control needed for a complex microservices environment.

#A. collector-proxy vs. eum-proxy

These are functional Envoy instances that act as Service Proxies.

  • Header-Based Routing: Envoy can look at specific HTTP headers or paths to decide exactly which version of a Collector pod should handle the request.
  • Retries and Timeouts: Envoy manages the "Resilience Logic." If a Collector pod is momentarily slow, Envoy can automatically retry the request to a different pod before the user's browser agent times out.
  • Circuit Breaking: If a cluster of Collector pods starts failing, Envoy "trips the circuit" to stop sending traffic to them, preventing a cascading failure across the entire EUM cloud.

#B. col-proxy-ctrl (The Control Plane)

This is the most critical theoretical component for a distributed system.

  • xDS APIs: In a modern architecture, you don't manually edit config files on 100 different proxies. Instead, Envoy uses a Control Plane (like Istio or a custom implementation) to discover its configuration.
  • Dynamic Updates: col-proxy-ctrl tells the proxies about new Collector pods as they spin up (Service Discovery). It allows the system to update routing rules in real-time without restarting the proxies.

#3. Why Have Both? (The Theory of "Proxy-in-Proxy")

You might wonder why we don't go straight from ALB to the Collector pods. The "Envoy Tier" adds several "Day 2" operational benefits:

  1. Observability: Envoy generates incredibly detailed metrics (success rates, latencies per route, 4xx/5xx counts) that the AWS ALB does not provide at the same level of granularity.
  2. Protocol Translation: The ALB might receive HTTP/2 from a modern browser, and Envoy can translate that to optimized gRPC or standard HTTP/1.1 for the internal Collector pods.
  3. Security (mTLS): While the ALB handles the public certificate, the Envoy tier can handle Internal Mutual TLS, ensuring that even if an attacker gets inside the VPC, they cannot spoof traffic between the proxy and the Collector pods.

#Summary of Traffic Flow

  1. Agent (e.g., a JS Browser Agent) sends a "Beacon."
  2. AWS ALB receives the HTTPS request, decrypts it, and identifies it as BRUM traffic.
  3. Envoy (collector-proxy) receives the request from the ALB. It checks the health of the Collector fleet and applies any specific routing rules (like "Canary" routing for a new feature).
  4. Collector Pod finally receives the validated, decrypted, and routed request to begin the Session Sampling logic.

#2.2 Key Platform Concepts for Interview

Beacon — The raw payload that an agent sends to the collector. Browser agents send HTTP POST beacons containing page load timing, AJAX results, JS errors, etc. Mobile agents send equivalent mobile-SDK beacons.

Collector — The front-door service in EUM Cloud that receives beacons, validates them, and converts them to internal record types. This is where your project lives.

BeaconProcessor — The processing component inside collector that parses beacon bytes into typed records (BrowserRecord, MobileRecord, etc.) and routes them through the validation and sampling pipeline.

Sessionizer — A dedicated downstream service that groups individual events/beacons into user sessions. It uses Kinesis stream partitioning by agentId to guarantee all events from the same session arrive at the same Kinesis consumer. This is fundamentally different from how collector works — collector is stateless across pods.

Decorator — Enriches session records with customer-configured labels and metadata before they are stored in the EuM Custom View Store.

AccountBeaconReporter — Reports valid-beacon counts and unsampled-beacon counts per app. This is what drives the EUM license consumption metrics.

#3. The Problem This Project Solved

#3.1 Business Problem

EUM customers pay for license units based on sessions processed. Many customers had no way to control how many sessions were being processed — every user session was ingested and counted toward their license. The requirement was to build session-based sampling so customers could set a target sampling percentage (e.g., 25%) and only process that fraction of their browser sessions.

#3.2 Technical Challenge — Why This Is Hard

Sampling sounds simple: flip a coin, decide yes or no. But three constraints make it genuinely hard in the EUM context:

Constraint 1 — Session coherence. The sampling unit must be a session, not an individual beacon. If you randomly sample individual beacons, you can keep a user's page-view event while dropping their related AJAX call and error event. That destroys session-level analysis. The entire session must be either fully sampled or fully dropped.

Constraint 2 — Distributed pods. Collector runs as multiple stateless pods behind a load balancer. There is no guarantee that two beacons from the same user session hit the same pod. If both pods independently decide whether to create a session, you get race conditions where two pods each think they are creating a brand new session for the same user — leading to duplicate sessions, inconsistent sampling decisions, and wrong license accounting.

Constraint 3 — Latency. Collector is a hot-path ingestion service handling millions of beacons per minute. Any distributed coordination must add minimal latency. Complex locking strategies that require multiple network round-trips are not acceptable.

#3.3 Why You Cannot Just Use Local Cache

The intuitive solution is to keep an in-memory map in each collector pod. Pod receives a beacon, checks the map, decides accept or reject.

This fails because the same session can hit different pods. Pod A creates session sess_12345 in its local map and accepts it. Moments later, Pod B receives another beacon from the same session but does not know about sess_12345. Pod B creates it again and also accepts it — you now have the same session counted twice.

You need a centralized, shared state store that all pods can coordinate through.

#2-Minute Version

Our EUM Browser Real User Monitoring system ingests massive browser traffic through distributed collector pods. Each beacon eventually contributes to metrics, event pipelines, and downstream sessionization. The business requirement was to give customers a way to save license consumption by sampling browser sessions rather than processing every single session.

The tricky part was that sampling had to happen early in collector so we could save compute, memory, queue traffic, and license consumption downstream. But collector is distributed — the same user's events can hit different pods. If every collector independently decides whether to create or sample a session, you get race conditions and inconsistent behavior.

My solution was to introduce a small distributed session layer in collector using Redis. Each collector uses Jedis to query and update Redis. The actual decision logic — whether this is a new session, an update to an existing session, a straggler, and whether it should be sampled — runs inside Redis through a Lua script. That gave us atomicity without introducing the multiple network round trips of distributed locking approaches like Redlock.

Sessions are stored with TTL so stale state is auto-cleaned. If Redis becomes unavailable, collectors can temporarily fall back to local processing so traffic continues flowing. Overall, this gave us consistent distributed sampling, lower latency, better license efficiency, and safer behavior at production scale.

#6.2 Inside the Redis Session Decision

When BrumSampler calls Redis via the Lua script, the following happens atomically in a single round-trip:

Input: KEYS[1..4] + ARGV[1..5]
    KEYS[1] = sampled_session:{appKey}:{sessionId}
    KEYS[2] = total_sessions_count:{appKey}
    KEYS[3] = accepted_sessions_count:{appKey}
    KEYS[4] = sampling_percentage:{appKey}
    ARGV[1] = session JSON (startTimeMs, endTimeMs, samplingStatus=false)
    ARGV[2] = startTimeMs
    ARGV[3] = endTimeMs
    ARGV[4] = timeoutMs
    ARGV[5] = targetSamplingPercentage

Decision flow inside Lua:
    if session key exists:
        if event time overlaps existing session window (with timeout):
            update session boundaries (min start, max end)
            refresh TTL
            return existing samplingStatus   <-- session continuity
        else:
            treat as new session (straggler replaces old session)
    else:
        if very first session for this app:
            always ACCEPT (bootstrap)
            init counters (total=1, accepted=1)
        else:
            current_pct = accepted_sessions / total_sessions * 100
            if current_pct < target_pct:  ACCEPT (incr both counters)
            if current_pct > target_pct:  REJECT (incr total only)
            if current_pct == target_pct: RANDOM (math.random(1..100) <= target?)
        
        set samplingStatus in session JSON
        set TTL = 600s + timeoutMs/1000
        return samplingStatus

#7. Redis Data Model — Complete Reference

This section is the comprehensive reference for everything stored in Redis as part of session-based sampling. All keys are namespaced per appKey to ensure complete isolation between applications sharing the same Redis instance.

#7.1 Key Naming Convention

Pattern:   {key_type}:{appKey}[:{sessionId}]
Separator: colon (:)
Scope:     All 4 keys are scoped per appKey
           Session key additionally scoped per sessionId

#7.2 Complete Key Inventory

Key 1 — Session Data Key (per session):
  Pattern:  sampled_session:{appKey}:{sessionId}
  Example:  sampled_session:EX-AAA-AUM:session_12345
  Type:     JSON (Redis JSON / ReJSON module)
  Purpose:  Stores session time boundaries + sampling decision
  TTL:      600s + (timeoutMillis / 1000)  →  auto-expires

Key 2 — Total Sessions Counter (per app):
  Pattern:  total_sessions_count:{appKey}
  Example:  total_sessions_count:EX-AAA-AUM
  Type:     Integer (Redis String, numeric)
  Purpose:  Running count of ALL sessions seen for this app (sampled + dropped)
  TTL:      None (persistent counter)
  Incremented by: Every new session creation

Key 3 — Accepted Sessions Counter (per app):
  Pattern:  accepted_sessions_count:{appKey}
  Example:  accepted_sessions_count:EX-AAA-AUM
  Type:     Integer (Redis String, numeric)
  Purpose:  Running count of only ACCEPTED sessions for this app
  TTL:      None (persistent counter)
  Incremented by: Only accepted new sessions

Key 4 — Sampling Percentage (per app):
  Pattern:  sampling_percentage:{appKey}
  Example:  sampling_percentage:EX-AAA-AUM
  Type:     Integer (Redis String, numeric)
  Purpose:  Target sampling percentage configured for this app (0-100)
  TTL:      None (persistent config)
  Set by:   First session creation for the app (cold start)

#7.3 Key Lifecycle

Cold Start (first session for app):
  WRITE: sampled_session:{appKey}:{sessionId}   ← created
  WRITE: total_sessions_count:{appKey}           ← SET 1
  WRITE: accepted_sessions_count:{appKey}        ← SET 1
  WRITE: sampling_percentage:{appKey}            ← SET {target}

Subsequent New Session:
  READ:  total_sessions_count:{appKey}
  READ:  accepted_sessions_count:{appKey}
  READ:  sampling_percentage:{appKey}
  WRITE: sampled_session:{appKey}:{sessionId}   ← created
  WRITE: total_sessions_count:{appKey}           ← INCR (always)
  WRITE: accepted_sessions_count:{appKey}        ← INCR (only if accepted)

Existing Session Update:
  READ:  sampled_session:{appKey}:{sessionId}   ← check existence + boundaries
  WRITE: sampled_session:{appKey}:{sessionId}   ← update startTimeMs / endTimeMs
  WRITE: sampled_session:{appKey}:{sessionId}   ← refresh TTL (EXPIRE)
  (counters are NOT touched for existing session updates)

KEYS[3] = "accepted_sessions_count:{appKey}"        -- accepted counter
KEYS[4] = "sampling_percentage:{appKey}"            -- target %

-- ARGV array: runtime values passed from Java (Jedis)
ARGV[1] = '{"startTimeMs":...,"endTimeMs":...,"samplingStatus":"false"}'
ARGV[2] = startTimeMs (long, epoch ms)
ARGV[3] = endTimeMs   (long, epoch ms)
ARGV[4] = timeoutMs   (long, ms — session gap tolerance)
ARGV[5] = targetSamplingPercentage (int, 0-100)

#8. Session Object Schema — Full Specification

#8.1 Session JSON Structure

The session JSON object stored at sampled_session:{appKey}:{sessionId} contains exactly 3 fields:

{
  "startTimeMs": 1698765432100,
  "endTimeMs":   1698765445200,
  "samplingStatus": "true"
}

#8.2 Field Definitions

FieldTypeDescriptionSet When
startTimeMslong (epoch ms)Earliest event start time seen in this sessionOn creation (from ARGV[2]); updated if a later beacon has an earlier start
endTimeMslong (epoch ms)Latest event end time seen in this sessionOn creation (from ARGV[3]); updated if a later beacon has a later end
samplingStatus"true" or "false" (string)Whether this session is accepted for processingOn creation (sampling decision); never changed after first set

#8.3 Field Behavior Detail

startTimeMs — always holds the minimum start across all beacons seen for this session. Each incoming beacon may report an earlier start (e.g., a background XHR that started before the page started). The Lua script extends the window backward if new_start < stored_start.

endTimeMs — always holds the maximum end across all beacons seen for this session. The Lua script extends the window forward if new_end > stored_end.

samplingStatus — immutable after first write. Once a session is decided as accepted ("true") or rejected ("false"), all subsequent beacons for that session return the same status. This is the session coherence guarantee. Note: stored as a string ("true" / "false"), not a JSON boolean.

#8.4 Schema Initialization (ARGV[1] template)

When a new session is first created, the Java code constructs the initial JSON and passes it as ARGV[1] with samplingStatus defaulted to "false". The Lua script then overwrites samplingStatus with the actual decision before returning:

{
  "startTimeMs": <from beacon>,
  "endTimeMs":   <from beacon>,
  "samplingStatus": "false"
}

The Lua script then calls JSON.SET key .samplingStatus "true" or "false" based on the sampling decision.

#8.5 Why the Schema Is Intentionally Minimal

Every beacon for every active session triggers a Redis read (EXISTS + JSON.GET) and potentially a write. At millions of beacons per minute, even small schema additions multiply into meaningful overhead:

  • Each additional field adds network payload per round-trip
  • Redis JSON parse/serialize cost scales with object size
  • Lua script complexity grows with more field accesses

Storing only startTimeMs, endTimeMs, samplingStatus keeps the per-operation cost as small as possible.

#8.6 TTL Calculation

TTL = sampled_session_retention_time_seconds + (timeoutMillis / 1000)
    = 600s                                   + 30s   (for 30s timeout config)
    = 630 seconds

Set via: EXPIRE sampled_session:{appKey}:{sessionId} 630
Refreshed on: every beacon update (existing session path)
Expires on:   no beacons received for 630 seconds

The timeoutMillis is the session gap tolerance — the maximum gap between beacons still considered part of the same session. Adding it to the base TTL ensures the session key does not expire during a legitimate gap.

#8.7 Counter Schema

The two counter keys store plain integers managed via Redis INCR:

total_sessions_count:{appKey}    → integer
accepted_sessions_count:{appKey} → integer

Sampling ratio = accepted / total * 100

Counter keys have no TTL. They persist across session expirations and accumulate for the lifetime of the application key in Redis.

#9. Lua Script — Atomic Session Decision

#9.1 Why Lua Instead of Redlock

This was the most important design decision in the project. Two approaches were considered:

Redlock (distributed locking):

  • Acquire lock on session key
  • Read current session state
  • Update session state
  • Release lock

Problem: this is 3+ network round trips to Redis per beacon. At millions of beacons per minute, this adds significant latency and load on the Redis cluster.

Lua Script in Redis:

  • Send one EVALSHA call with all inputs
  • Redis executes the full read-check-update-decide workflow atomically server-side
  • One network round trip, total

Result: same atomicity guarantee, fewer round trips, lower latency, simpler control flow.

Interview framing: "Same correctness goal, fewer round trips, lower latency, simpler operationally. In a collector path doing millions of operations per minute, this difference matters."

#9.2 Atomicity Guarantee

Redis runs Lua scripts atomically. No other command can execute while a Lua script is running. This means:

  • Two collectors submitting simultaneous requests for sess_12345 will be serialized by Redis
  • The second caller sees the state written by the first — no duplicate session creation
  • No distributed locks needed at the application layer

#9.3 Core Script Logic (Pseudocode)

-- Check if session already exists
local session_exists = redis.call('EXISTS', KEYS[1])

if session_exists == 1 then
    -- Existing session: check time overlap
    local stored_start = JSON.GET(KEYS[1], '.startTimeMs')
    local stored_end   = JSON.GET(KEYS[1], '.endTimeMs')
    local new_start    = ARGV[2]
    local new_end      = ARGV[3]
    local timeout      = ARGV[4]
    
    -- Time overlap check
    if new_end >= (stored_start - timeout) and new_start <= (stored_end + timeout) then
        -- Same session: update boundaries, return existing decision
        if stored_start > new_start then JSON.SET(KEYS[1], '.startTimeMs', new_start) end
        if stored_end   < new_end   then JSON.SET(KEYS[1], '.endTimeMs',   new_end)   end
        redis.call('EXPIRE', KEYS[1], TTL)
        return JSON.GET(KEYS[1], '.samplingStatus')
    end
end

-- New session: create entry
JSON.SET(KEYS[1], '.', ARGV[1])

-- Initialize or read counters
local accepted = tonumber(redis.call('GET', KEYS[3])) or 0
local total    = tonumber(redis.call('GET', KEYS[2])) or 0
local target   = tonumber(ARGV[5])

if total == 0 then
    -- First ever session for this app: always accept
    redis.call('SET', KEYS[4], target)
    redis.call('SET', KEYS[2], 1)
    redis.call('SET', KEYS[3], 1)
    JSON.SET(KEYS[1], '.samplingStatus', 'true')
else
    local current_pct = accepted / total * 100
    redis.call('INCR', KEYS[2])  -- always increment total

    if current_pct < target then
        -- Under target: ACCEPT
        redis.call('INCR', KEYS[3])
        JSON.SET(KEYS[1], '.samplingStatus', 'true')
    elseif current_pct > target then
        -- Over target: REJECT (do NOT increment accepted)
        JSON.SET(KEYS[1], '.samplingStatus', 'false')
    else
        -- At target: random decision
        local rand = math.random(100)
        if rand <= target then
            redis.call('INCR', KEYS[3])
            JSON.SET(KEYS[1], '.samplingStatus', 'true')
        else
            JSON.SET(KEYS[1], '.samplingStatus', 'false')
        end
    end
end

-- Set TTL and return
redis.call('EXPIRE', KEYS[1], 600 + tonumber(ARGV[4]) / 1000)
return JSON.GET(KEYS[1], '.samplingStatus')

#12. Design Decisions and Tradeoffs

#Why Redis?

Redis provides:

  • Sub-millisecond read/write latency at scale
  • Built-in TTL support — automatic key expiration
  • Server-side Lua scripting with atomic execution guarantee
  • Wide operational familiarity in production infrastructure

Alternative considered: an in-memory distributed cache like Memcached — but it lacks Lua scripting support, making the atomic session-decision pattern impossible without application-level locking.

#Why Jedis?

The EUM collector is a Java service. Jedis is a mature, battle-tested Java Redis client with:

  • Built-in connection pooling (JedisPool / JedisPooled)
  • Configurable read/write timeouts
  • evalsha support for Lua script pre-loading
  • Low dependency footprint

Pooling is important because many beacon-processing threads need to hit Redis concurrently. A single shared connection would serialize all requests.

#Why Lua Instead of Redlock?

RedlockLua Script
Network round trips3+ (lock acquire, read, write, lock release)1 (single EVALSHA)
AtomicityVia distributed lockingRedis native atomic execution
ComplexityLock management, deadlock handling, retry logicSingle script file
Latency impactHighMinimal
Failure modeLock can expire mid-operationScript completes or fails atomically

In a latency-sensitive ingestion path, the 1-call Lua approach wins clearly.

#Why Not Use Collector Local Cache Only?

Collector is a stateless, horizontally scaled service. Multiple pods can receive events for the same session simultaneously. A local in-memory map only knows about events processed by that specific pod. There is no cross-pod shared state, so you cannot guarantee consistent sampling decisions for the same session.

#Why TTL Instead of Explicit Cleanup?

Explicit cleanup (a scheduled job that deletes old sessions) adds:

  • Operational complexity
  • Risk of premature deletion for active sessions
  • Need for distributed coordination on cleanup itself

TTL aligns cleanup with the session lifecycle automatically. When a session is no longer active and the TTL expires, Redis removes the key. No code needed.

#Why Keep the Session Object Small?

Every beacon for every session triggers a Redis read. At millions of beacons per minute, even small increases in payload size multiply into meaningful network, memory, and CPU overhead. Storing only startTimeMs, endTimeMs, and samplingStatus keeps the Redis footprint minimal.

#Fallback Tradeoff

When Redis becomes unavailable:

  • Strict centralized sampling consistency is impossible
  • Collectors fall back to local processing to avoid blocking ingestion
  • Some sessions may be double-counted or not sampled correctly during the outage window
  • Availability is prioritized over perfect sampling consistency

This is the correct tradeoff for a front-door ingestion service where data loss (blocking beacons) is worse than temporary sampling inaccuracy.

#16. Interview Q and A

#Q: Why did you sample sessions instead of individual browser beacons?

A browser session is the meaningful analytical unit in BRUM. If you sample individual beacons independently, you can keep some page-view events while dropping related AJAX calls, resources, or follow-up interactions. That breaks session integrity and distorts user-behavior analytics.

Session-based sampling preserves the coherence of the user journey. Either the whole session is accepted or the whole session is rejected, which keeps metrics and investigations meaningful.

#Q: Why couldn't you use an in-memory map in collector?

Collector runs as multiple stateless pods. The same user's events can reach different pods. A local cache only knows about events seen by that particular pod. Two pods could each independently create session_12345, resulting in duplicate session records and inconsistent sampling decisions.

Redis solves this by acting as a shared, centralized coordination layer that every collector pod queries and updates.

#Q: Why Lua script over Redlock?

The collector path is latency-sensitive. Redlock requires multiple calls: acquire lock, read/update session data, release lock. Lua lets Redis execute the entire decision path atomically in one server-side script. That means fewer network round trips, lower latency, and simpler control flow.

We achieved the same atomicity goal with better performance characteristics for our workload.

#Q: How did you handle the race condition exactly?

Lua scripts run atomically inside Redis. No other command can execute while a Lua script is running. So even if two collectors submit near-simultaneous requests for the same session key, Redis serializes execution. The first script invocation creates/updates the session, and the second sees the updated state rather than racing on stale reads.

This eliminated the duplicate-session creation issue that occurred when multiple collectors independently tried to create the same session.

#Q: What happens if Redis goes down?

The design includes fallback local processing so collectors can continue ingesting traffic rather than blocking or losing data. This prioritizes availability in a front-door ingestion service.

Perfect global sampling consistency may be temporarily reduced during a Redis outage, but the system continues operating and switches back to Redis when connectivity is restored.

#Q: How does the algorithm stay near the target sampling percentage?

The Lua script maintains per-app counters in Redis: total sessions seen and accepted sessions. For each new session, it computes accepted / total * 100 and compares with the target.

  • Below target: accept more sessions
  • Above target: reject more sessions
  • Exactly at target: randomized decision to avoid deterministic bias

This creates a self-correcting feedback loop rather than a naive one-time random decision. Over time the system converges on the configured target percentage.

#Q: Why is TTL important here?

Redis is serving as a live coordination cache, not a permanent session history. Sessions that become inactive should disappear automatically. TTL prevents memory bloat and eliminates the need for a separate cleanup process. It aligns key lifetime with the session's natural lifetime.

#Q: What was the hardest design decision?

Deciding how to resolve the distributed race condition without adding too much latency to the collector hot path. That decision directly affected correctness, performance, and production safety.

Evaluating Redlock versus Lua scripting was the key tradeoff. Lua was chosen because it preserved atomicity while avoiding the multiple extra Redis API calls that Redlock would require at scale.

#Q: What is the session continuity logic?

When a beacon arrives for an existing session key, the Lua script checks for time overlap:

new_end >= stored_start - timeout  AND  new_start <= stored_end + timeout

If this overlap condition holds, the new event is treated as part of the existing session. The script updates the session's start and end boundaries (taking the earliest start and latest end) and refreshes the TTL. The original sampling decision is returned unchanged — ensuring consistency for all events in that session.

If the time ranges do not overlap (a straggler from a completely different time period), the session is treated as a new session and goes through the full sampling decision.

#Q: What happens if the first session for a new app arrives?

The first session is always accepted. This is the bootstrap case. The script detects total_sessions_count = 0, sets counters to (total=1, accepted=1), stores the target sampling percentage, and marks the session as accepted. This prevents a cold-start rejection where the very first session could be dropped before any ratio exists.

#Q: How does sampling for MRUM differ from BRUM?

The underlying algorithm is the same — same Redis key structure, same Lua script logic, same per-app counters and self-correcting behavior. The difference is that MrumSampler.sampleMobileRecord() extracts session information from a mobile beacon format (iOS/Android SDK) rather than a browser beacon format. The feature flag is mrumSamplingEnabled (separate from sessionBasedSamplingEnabled for BRUM).

#Q: What exactly is stored in Redis and why?

Four key types per appKey: (1) the session JSON object per active sessionId holding time boundaries and sampling decision, (2) a total sessions counter, (3) an accepted sessions counter, and (4) the target sampling percentage. The counter pair drives the self-correcting algorithm. The session object enables session continuity — once a decision is made, every subsequent beacon for that session reads and returns the same status without recalculating.

#Q: Why is samplingStatus stored as a string rather than a JSON boolean?

Redis JSON module interop with Lua returns values as strings. Storing "true" and "false" as string values avoids type coercion issues when reading back from Redis via JSON.GET inside Lua and when parsing the return value back in Java via Jedis.

### 2. Beacon Data — Where Does It Go?
## A browser beacon is an HTTP POST with a binary or JSON payload. Here is its lifecycle:

```
Step 1: Browser JS agent sends HTTP POST beacon
        Payload example:
        {
          "sessionId":   "sess_abc123",
          "appKey":      "EX-AAA-AUM",
          "startTime":   1698765432100,
          "endTime":     1698765445200,
          "pageUrl":     "https://example.com/checkout",
          "ajaxCalls":   [...],
          "jsErrors":    [...],
          "resourceTimings": [...]
        }

Step 2: HTTP Servlet in Collector receives it
        → Calls BeaconProcessor.processBeacon(rawBytes)

Step 3: BeaconProcessor parses raw bytes → BrowserRecord Java object
        BrowserRecord {
          sessionId:   "sess_abc123"
          appKey:      "EX-AAA-AUM"
          startTimeMs: 1698765432100
          endTimeMs:   1698765445200
          ...other fields
        }
        ↑ This object lives on JVM heap. It is NOT written to disk or DB.

Step 4: BrumSampler.sampleBrowserRecord(record) is called
        → Reads/writes ONLY to Redis (my project)
        → Returns true (accept) or false (drop)

Step 5a: If ACCEPTED → push BrowserRecord to Kinesis stream
         → downstream Sessionizer picks it up
         → eventually lands in EuM Custom View Store

Step 5b: If REJECTED → record is discarded from memory
         → reportUnsampledBeaconCount() called
         → nothing written anywhere (beacon is dropped)
Beacon data itself is never persisted by Collector. It is either forwarded downstream or dropped.
```

Before a single metric is captured, the Browser Agent (JS agent) must be loaded into the user's browser.

Regional Deployment: The JS agent is hosted on CDNs (like CloudFront or Akamai) across multiple global regions. Version Management: CDNs allow for "staged rollouts" of agent updates. Different regions can host different versions of the agent, allowing for canary testing of new collection features.

### 2. Data Collection: The Beacon API
Once the agent is running in the user's browser, it hooks into the Beacon API and other browser performance interfaces.

The "Collection" Phase: The agent gathers page load timings, AJAX calls, and JS errors. It packages this into a binary or JSON payload known as a "Beacon."

The Transmission Phase: Instead of sending data back to the CDN (which is primarily for delivering content), the agent sends an HTTP POST request directly to the EUM Cloud Ingress.

## 1. The Entry Point: AWS Application Load Balancers (ALB)

The ALB is the "External Facing" tier. Its primary job is **Layer 7 (HTTP/HTTPS) termination** and initial traffic segregation.

* **Traffic Segregation:** By using different ALBs for different traffic types (BRUM vs. MRUM vs. ThousandEye), the system achieves **fault isolation**. If the `Collector-proxy ALB` is overwhelmed by a spike in Browser traffic, the `TE-Collector ALB` remains unaffected, ensuring ThousandEye monitoring persists.
* **SSL/TLS Termination:** The ALBs handle the compute-heavy process of decrypting HTTPS traffic from the public internet. This offloads CPU cycles from the internal pods.
* **Scalability:** ALBs are managed services that scale automatically to handle throughput spikes. They act as the "shock absorbers" for the EUM cloud.

---

## 2. The Intelligence Layer: Envoy Proxy Tiers

Once the ALB accepts a request, it passes it to **Envoy**, a high-performance C++ distributed proxy. While the ALB is great at basic load balancing, Envoy provides the **fine-grained control** needed for a complex microservices environment.

### A. collector-proxy vs. eum-proxy

These are functional Envoy instances that act as **Service Proxies**.

* **Header-Based Routing:** Envoy can look at specific HTTP headers or paths to decide exactly which version of a Collector pod should handle the request.
* **Retries and Timeouts:** Envoy manages the "Resilience Logic." If a Collector pod is momentarily slow, Envoy can automatically retry the request to a different pod before the user's browser agent times out.
* **Circuit Breaking:** If a cluster of Collector pods starts failing, Envoy "trips the circuit" to stop sending traffic to them, preventing a cascading failure across the entire EUM cloud.

### B. col-proxy-ctrl (The Control Plane)

This is the most critical theoretical component for a distributed system.

* **xDS APIs:** In a modern architecture, you don't manually edit config files on 100 different proxies. Instead, Envoy uses a **Control Plane** (like Istio or a custom implementation) to discover its configuration.
* **Dynamic Updates:** `col-proxy-ctrl` tells the proxies about new Collector pods as they spin up (Service Discovery). It allows the system to update routing rules in real-time without restarting the proxies.

---

## 3. Why Have Both? (The Theory of "Proxy-in-Proxy")

You might wonder why we don't go straight from ALB to the Collector pods. The "Envoy Tier" adds several "Day 2" operational benefits:

1. **Observability:** Envoy generates incredibly detailed metrics (success rates, latencies per route, 4xx/5xx counts) that the AWS ALB does not provide at the same level of granularity.
2. **Protocol Translation:** The ALB might receive HTTP/2 from a modern browser, and Envoy can translate that to optimized gRPC or standard HTTP/1.1 for the internal Collector pods.
3. **Security (mTLS):** While the ALB handles the public certificate, the Envoy tier can handle **Internal Mutual TLS**, ensuring that even if an attacker gets inside the VPC, they cannot spoof traffic between the proxy and the Collector pods.

---

### Summary of Traffic Flow

1. **Agent** (e.g., a JS Browser Agent) sends a "Beacon."
2. **AWS ALB** receives the HTTPS request, decrypts it, and identifies it as `BRUM` traffic.
3. **Envoy (`collector-proxy`)** receives the request from the ALB. It checks the health of the Collector fleet and applies any specific routing rules (like "Canary" routing for a new feature).
4. **Collector Pod** finally receives the validated, decrypted, and routed request to begin the [Session Sampling logic](https://github.com/abhishekgoyal17/Level_up/blob/main/01_EUM_session_sampling.md#6-architecture-and-flow).

### 2.2 Key Platform Concepts for Interview

**Beacon** — The raw payload that an agent sends to the collector. Browser agents send HTTP POST beacons containing page load timing, AJAX results, JS errors, etc. Mobile agents send equivalent mobile-SDK beacons.

**Collector** — The front-door service in EUM Cloud that receives beacons, validates them, and converts them to internal record types. This is where your project lives.

**BeaconProcessor** — The processing component inside collector that parses beacon bytes into typed records (`BrowserRecord`, `MobileRecord`, etc.) and routes them through the validation and sampling pipeline.

**Sessionizer** — A dedicated downstream service that groups individual events/beacons into user sessions. It uses Kinesis stream partitioning by `agentId` to guarantee all events from the same session arrive at the same Kinesis consumer. This is fundamentally different from how collector works — collector is stateless across pods.

**Decorator** — Enriches session records with customer-configured labels and metadata before they are stored in the EuM Custom View Store.

**AccountBeaconReporter** — Reports valid-beacon counts and unsampled-beacon counts per app. This is what drives the EUM license consumption metrics.

## 3. The Problem This Project Solved

### 3.1 Business Problem

EUM customers pay for license units based on sessions processed. Many customers had no way to control how many sessions were being processed — every user session was ingested and counted toward their license. The requirement was to build **session-based sampling** so customers could set a target sampling percentage (e.g., 25%) and only process that fraction of their browser sessions.

### 3.2 Technical Challenge — Why This Is Hard

Sampling sounds simple: flip a coin, decide yes or no. But three constraints make it genuinely hard in the EUM context:

**Constraint 1 — Session coherence.** The sampling unit must be a session, not an individual beacon. If you randomly sample individual beacons, you can keep a user's page-view event while dropping their related AJAX call and error event. That destroys session-level analysis. The entire session must be either fully sampled or fully dropped.

**Constraint 2 — Distributed pods.** Collector runs as multiple stateless pods behind a load balancer. There is no guarantee that two beacons from the same user session hit the same pod. If both pods independently decide whether to create a session, you get race conditions where two pods each think they are creating a brand new session for the same user — leading to duplicate sessions, inconsistent sampling decisions, and wrong license accounting.

**Constraint 3 — Latency.** Collector is a hot-path ingestion service handling millions of beacons per minute. Any distributed coordination must add minimal latency. Complex locking strategies that require multiple network round-trips are not acceptable.

### 3.3 Why You Cannot Just Use Local Cache

The intuitive solution is to keep an in-memory map in each collector pod. Pod receives a beacon, checks the map, decides accept or reject.

This fails because the same session can hit different pods. Pod A creates session `sess_12345` in its local map and accepts it. Moments later, Pod B receives another beacon from the same session but does not know about `sess_12345`. Pod B creates it again and also accepts it — you now have the same session counted twice.

You need a centralized, shared state store that all pods can coordinate through.

### 2-Minute Version

Our EUM Browser Real User Monitoring system ingests massive browser traffic through distributed collector pods. Each beacon eventually contributes to metrics, event pipelines, and downstream sessionization. The business requirement was to give customers a way to save license consumption by sampling browser sessions rather than processing every single session.

The tricky part was that sampling had to happen **early in collector** so we could save compute, memory, queue traffic, and license consumption downstream. But collector is distributed — the same user's events can hit different pods. If every collector independently decides whether to create or sample a session, you get race conditions and inconsistent behavior.

Sessions are stored with TTL so stale state is auto-cleaned. If Redis becomes unavailable, collectors can temporarily fall back to local processing so traffic continues flowing. Overall, this gave us consistent distributed sampling, lower latency, better license efficiency, and safer behavior at production scale.

### 6.2 Inside the Redis Session Decision

When BrumSampler calls Redis via the Lua script, the following happens atomically in a single round-trip:

```
Input: KEYS[1..4] + ARGV[1..5]
    KEYS[1] = sampled_session:{appKey}:{sessionId}
    KEYS[2] = total_sessions_count:{appKey}
    KEYS[3] = accepted_sessions_count:{appKey}
    KEYS[4] = sampling_percentage:{appKey}
    ARGV[1] = session JSON (startTimeMs, endTimeMs, samplingStatus=false)
    ARGV[2] = startTimeMs
    ARGV[3] = endTimeMs
    ARGV[4] = timeoutMs
    ARGV[5] = targetSamplingPercentage

Decision flow inside Lua:
    if session key exists:
        if event time overlaps existing session window (with timeout):
            update session boundaries (min start, max end)
            refresh TTL
            return existing samplingStatus   <-- session continuity
        else:
            treat as new session (straggler replaces old session)
    else:
        if very first session for this app:
            always ACCEPT (bootstrap)
            init counters (total=1, accepted=1)
        else:
            current_pct = accepted_sessions / total_sessions * 100
            if current_pct < target_pct:  ACCEPT (incr both counters)
            if current_pct > target_pct:  REJECT (incr total only)
            if current_pct == target_pct: RANDOM (math.random(1..100) <= target?)
        
        set samplingStatus in session JSON
        set TTL = 600s + timeoutMs/1000
        return samplingStatus
```

### 6.3 Sampling Decision Engine — Object Model

The Sampling Decision Engine is triggered inside `BeaconProcessor` and classifies incoming records:

| Record Type | Condition | Sampler Invoked |
|---|---|---|
| `BrowserRecord` | `sessionBasedSamplingEnabled = true` | `BrumSampler.sampleBrowserRecord()` |
| `MobileRecord` | `mrumSamplingEnabled = true` | `MrumSampler.sampleMobileRecord()` |
| Other (`ARecord`, etc.) | `samplingRequired = false` | Bypasses sampling entirely |

## 8. Session Object Schema — Full Specification

### 8.1 Session JSON Structure

The session JSON object stored at `sampled_session:{appKey}:{sessionId}` contains exactly 3 fields:

```json
{
  "startTimeMs": 1698765432100,
  "endTimeMs":   1698765445200,
  "samplingStatus": "true"
}
```

### 8.2 Field Definitions

| Field | Type | Description | Set When |
|---|---|---|---|
| `startTimeMs` | `long` (epoch ms) | Earliest event start time seen in this session | On creation (from ARGV[2]); updated if a later beacon has an earlier start |
| `endTimeMs` | `long` (epoch ms) | Latest event end time seen in this session | On creation (from ARGV[3]); updated if a later beacon has a later end |
| `samplingStatus` | `"true"` or `"false"` (string) | Whether this session is accepted for processing | On creation (sampling decision); never changed after first set |

### 8.3 Field Behavior Detail

**`startTimeMs`** — always holds the minimum start across all beacons seen for this session. Each incoming beacon may report an earlier start (e.g., a background XHR that started before the page started). The Lua script extends the window backward if `new_start < stored_start`.

**`endTimeMs`** — always holds the maximum end across all beacons seen for this session. The Lua script extends the window forward if `new_end > stored_end`.

**`samplingStatus`** — immutable after first write. Once a session is decided as accepted (`"true"`) or rejected (`"false"`), all subsequent beacons for that session return the same status. This is the session coherence guarantee. Note: stored as a string (`"true"` / `"false"`), not a JSON boolean.

### 8.4 Schema Initialization (ARGV[1] template)

When a new session is first created, the Java code constructs the initial JSON and passes it as ARGV[1] with `samplingStatus` defaulted to `"false"`. The Lua script then overwrites `samplingStatus` with the actual decision before returning:

```json
{
  "startTimeMs": <from beacon>,
  "endTimeMs":   <from beacon>,
  "samplingStatus": "false"
}
```

The Lua script then calls `JSON.SET key .samplingStatus "true"` or `"false"` based on the sampling decision.

### 8.5 Why the Schema Is Intentionally Minimal

Every beacon for every active session triggers a Redis read (EXISTS + JSON.GET) and potentially a write. At millions of beacons per minute, even small schema additions multiply into meaningful overhead:

- Each additional field adds network payload per round-trip
- Redis JSON parse/serialize cost scales with object size
- Lua script complexity grows with more field accesses

Storing only `startTimeMs`, `endTimeMs`, `samplingStatus` keeps the per-operation cost as small as possible.

### 8.6 TTL Calculation

```
TTL = sampled_session_retention_time_seconds + (timeoutMillis / 1000)
    = 600s                                   + 30s   (for 30s timeout config)
    = 630 seconds

Set via: EXPIRE sampled_session:{appKey}:{sessionId} 630
Refreshed on: every beacon update (existing session path)
Expires on:   no beacons received for 630 seconds
```

The `timeoutMillis` is the session gap tolerance — the maximum gap between beacons still considered part of the same session. Adding it to the base TTL ensures the session key does not expire during a legitimate gap.

### 8.7 Counter Schema

The two counter keys store plain integers managed via Redis INCR:

```
total_sessions_count:{appKey}    → integer
accepted_sessions_count:{appKey} → integer

Sampling ratio = accepted / total * 100
```

Counter keys have no TTL. They persist across session expirations and accumulate for the lifetime of the application key in Redis.

## 9. Lua Script — Atomic Session Decision

### 9.1 Why Lua Instead of Redlock

This was the most important design decision in the project. Two approaches were considered:

**Redlock (distributed locking):**
- Acquire lock on session key
- Read current session state
- Update session state
- Release lock

Problem: this is 3+ network round trips to Redis per beacon. At millions of beacons per minute, this adds significant latency and load on the Redis cluster.

**Lua Script in Redis:**
- Send one `EVALSHA` call with all inputs
- Redis executes the full read-check-update-decide workflow atomically server-side
- One network round trip, total

Result: same atomicity guarantee, fewer round trips, lower latency, simpler control flow.

> **Interview framing:** "Same correctness goal, fewer round trips, lower latency, simpler operationally. In a collector path doing millions of operations per minute, this difference matters."

### 9.2 Atomicity Guarantee

Redis runs Lua scripts atomically. No other command can execute while a Lua script is running. This means:
- Two collectors submitting simultaneous requests for `sess_12345` will be serialized by Redis
- The second caller sees the state written by the first — no duplicate session creation
- No distributed locks needed at the application layer

### 9.3 Core Script Logic (Pseudocode)

```lua
-- Check if session already exists
local session_exists = redis.call('EXISTS', KEYS[1])

if session_exists == 1 then
    -- Existing session: check time overlap
    local stored_start = JSON.GET(KEYS[1], '.startTimeMs')
    local stored_end   = JSON.GET(KEYS[1], '.endTimeMs')
    local new_start    = ARGV[2]
    local new_end      = ARGV[3]
    local timeout      = ARGV[4]
    
    -- Time overlap check
    if new_end >= (stored_start - timeout) and new_start <= (stored_end + timeout) then
        -- Same session: update boundaries, return existing decision
        if stored_start > new_start then JSON.SET(KEYS[1], '.startTimeMs', new_start) end
        if stored_end   < new_end   then JSON.SET(KEYS[1], '.endTimeMs',   new_end)   end
        redis.call('EXPIRE', KEYS[1], TTL)
        return JSON.GET(KEYS[1], '.samplingStatus')
    end
end

-- New session: create entry
JSON.SET(KEYS[1], '.', ARGV[1])

-- Initialize or read counters
local accepted = tonumber(redis.call('GET', KEYS[3])) or 0
local total    = tonumber(redis.call('GET', KEYS[2])) or 0
local target   = tonumber(ARGV[5])

if total == 0 then
    -- First ever session for this app: always accept
    redis.call('SET', KEYS[4], target)
    redis.call('SET', KEYS[2], 1)
    redis.call('SET', KEYS[3], 1)
    JSON.SET(KEYS[1], '.samplingStatus', 'true')
else
    local current_pct = accepted / total * 100
    redis.call('INCR', KEYS[2])  -- always increment total

    if current_pct < target then
        -- Under target: ACCEPT
        redis.call('INCR', KEYS[3])
        JSON.SET(KEYS[1], '.samplingStatus', 'true')
    elseif current_pct > target then
        -- Over target: REJECT (do NOT increment accepted)
        JSON.SET(KEYS[1], '.samplingStatus', 'false')
    else
        -- At target: random decision
        local rand = math.random(100)
        if rand <= target then
            redis.call('INCR', KEYS[3])
            JSON.SET(KEYS[1], '.samplingStatus', 'true')
        else
            JSON.SET(KEYS[1], '.samplingStatus', 'false')
        end
    end
end

-- Set TTL and return
redis.call('EXPIRE', KEYS[1], 600 + tonumber(ARGV[4]) / 1000)
return JSON.GET(KEYS[1], '.samplingStatus')
```

## 11. Where This Fits in Collector Flow

Sampling was placed **intentionally early** in the collector pipeline:

```
Browser Beacon hits collector
        |
        v
BeaconProcessor: parse + validate
        |
        v
[SAMPLING GATE — Redis + Jedis + Lua]  <-- YOUR PROJECT
        |
  ACCEPT        REJECT
    |               |
    v               v
Record Queue    Record Dropped
(only sampled   reportUnsampled()
 sessions)
    |
    v
Downstream:
  Metrics Pipeline
  Event Grouper
  Crash Grouper
  Sessionizer (Kinesis)
  Decorator
```

**Why early placement matters:** If you sample too late, you still pay for:
- Record creation and object allocation
- Validation CPU cycles
- Queue memory and capacity
- SQS/Kinesis message transmission
- Downstream pipeline execution
- License accounting for sessions that will be dropped anyway

Sampling early in collector means unsampled sessions are dropped **before** they consume any of those resources.

## 12. Design Decisions and Tradeoffs

### Why Redis?

Redis provides:
- Sub-millisecond read/write latency at scale
- Built-in TTL support — automatic key expiration
- Server-side Lua scripting with atomic execution guarantee
- Wide operational familiarity in production infrastructure

Alternative considered: an in-memory distributed cache like Memcached — but it lacks Lua scripting support, making the atomic session-decision pattern impossible without application-level locking.

### Why Jedis?

The EUM collector is a Java service. Jedis is a mature, battle-tested Java Redis client with:
- Built-in connection pooling (`JedisPool` / `JedisPooled`)
- Configurable read/write timeouts
- `evalsha` support for Lua script pre-loading
- Low dependency footprint

Pooling is important because many beacon-processing threads need to hit Redis concurrently. A single shared connection would serialize all requests.

### Why Lua Instead of Redlock?

| | Redlock | Lua Script |
|---|---|---|
| Network round trips | 3+ (lock acquire, read, write, lock release) | 1 (single EVALSHA) |
| Atomicity | Via distributed locking | Redis native atomic execution |
| Complexity | Lock management, deadlock handling, retry logic | Single script file |
| Latency impact | High | Minimal |
| Failure mode | Lock can expire mid-operation | Script completes or fails atomically |

In a latency-sensitive ingestion path, the 1-call Lua approach wins clearly.

### Why Not Use Collector Local Cache Only?

Collector is a stateless, horizontally scaled service. Multiple pods can receive events for the same session simultaneously. A local in-memory map only knows about events processed by that specific pod. There is no cross-pod shared state, so you cannot guarantee consistent sampling decisions for the same session.

### Why TTL Instead of Explicit Cleanup?

Explicit cleanup (a scheduled job that deletes old sessions) adds:
- Operational complexity
- Risk of premature deletion for active sessions
- Need for distributed coordination on cleanup itself

TTL aligns cleanup with the session lifecycle automatically. When a session is no longer active and the TTL expires, Redis removes the key. No code needed.

### Why Keep the Session Object Small?

Every beacon for every session triggers a Redis read. At millions of beacons per minute, even small increases in payload size multiply into meaningful network, memory, and CPU overhead. Storing only `startTimeMs`, `endTimeMs`, and `samplingStatus` keeps the Redis footprint minimal.

### Fallback Tradeoff

When Redis becomes unavailable:
- Strict centralized sampling consistency is impossible
- Collectors fall back to local processing to avoid blocking ingestion
- Some sessions may be double-counted or not sampled correctly during the outage window
- Availability is prioritized over perfect sampling consistency

This is the correct tradeoff for a front-door ingestion service where data loss (blocking beacons) is worse than temporary sampling inaccuracy.

## 16. Interview Q and A

### Q: Why did you sample sessions instead of individual browser beacons?

A browser session is the meaningful analytical unit in BRUM. If you sample individual beacons independently, you can keep some page-view events while dropping related AJAX calls, resources, or follow-up interactions. That breaks session integrity and distorts user-behavior analytics.

Session-based sampling preserves the coherence of the user journey. Either the whole session is accepted or the whole session is rejected, which keeps metrics and investigations meaningful.

### Q: Why couldn't you use an in-memory map in collector?

Collector runs as multiple stateless pods. The same user's events can reach different pods. A local cache only knows about events seen by that particular pod. Two pods could each independently create `session_12345`, resulting in duplicate session records and inconsistent sampling decisions.

Redis solves this by acting as a shared, centralized coordination layer that every collector pod queries and updates.

### Q: Why Lua script over Redlock?

The collector path is latency-sensitive. Redlock requires multiple calls: acquire lock, read/update session data, release lock. Lua lets Redis execute the entire decision path atomically in one server-side script. That means fewer network round trips, lower latency, and simpler control flow.

We achieved the same atomicity goal with better performance characteristics for our workload.

### Q: How did you handle the race condition exactly?

Lua scripts run atomically inside Redis. No other command can execute while a Lua script is running. So even if two collectors submit near-simultaneous requests for the same session key, Redis serializes execution. The first script invocation creates/updates the session, and the second sees the updated state rather than racing on stale reads.

This eliminated the duplicate-session creation issue that occurred when multiple collectors independently tried to create the same session.

### Q: What happens if Redis goes down?

The design includes fallback local processing so collectors can continue ingesting traffic rather than blocking or losing data. This prioritizes availability in a front-door ingestion service.

Perfect global sampling consistency may be temporarily reduced during a Redis outage, but the system continues operating and switches back to Redis when connectivity is restored.

### Q: How does the algorithm stay near the target sampling percentage?

The Lua script maintains per-app counters in Redis: total sessions seen and accepted sessions. For each new session, it computes `accepted / total * 100` and compares with the target.

- Below target: accept more sessions
- Above target: reject more sessions
- Exactly at target: randomized decision to avoid deterministic bias

This creates a self-correcting feedback loop rather than a naive one-time random decision. Over time the system converges on the configured target percentage.

### Q: Why is TTL important here?

Redis is serving as a live coordination cache, not a permanent session history. Sessions that become inactive should disappear automatically. TTL prevents memory bloat and eliminates the need for a separate cleanup process. It aligns key lifetime with the session's natural lifetime.

### Q: What was the hardest design decision?

Deciding how to resolve the distributed race condition without adding too much latency to the collector hot path. That decision directly affected correctness, performance, and production safety.

Evaluating Redlock versus Lua scripting was the key tradeoff. Lua was chosen because it preserved atomicity while avoiding the multiple extra Redis API calls that Redlock would require at scale.

### Q: What is the session continuity logic?

When a beacon arrives for an existing session key, the Lua script checks for time overlap:

```
new_end >= stored_start - timeout  AND  new_start <= stored_end + timeout
```

If this overlap condition holds, the new event is treated as part of the existing session. The script updates the session's start and end boundaries (taking the earliest start and latest end) and refreshes the TTL. The original sampling decision is returned unchanged — ensuring consistency for all events in that session.

If the time ranges do not overlap (a straggler from a completely different time period), the session is treated as a new session and goes through the full sampling decision.

### Q: What happens if the first session for a new app arrives?

The first session is always accepted. This is the bootstrap case. The script detects `total_sessions_count = 0`, sets counters to `(total=1, accepted=1)`, stores the target sampling percentage, and marks the session as accepted. This prevents a cold-start rejection where the very first session could be dropped before any ratio exists.

### Q: How does sampling for MRUM differ from BRUM?

The underlying algorithm is the same — same Redis key structure, same Lua script logic, same per-app counters and self-correcting behavior. The difference is that `MrumSampler.sampleMobileRecord()` extracts session information from a mobile beacon format (iOS/Android SDK) rather than a browser beacon format. The feature flag is `mrumSamplingEnabled` (separate from `sessionBasedSamplingEnabled` for BRUM).

### Q: What exactly is stored in Redis and why?

Four key types per `appKey`: (1) the session JSON object per active `sessionId` holding time boundaries and sampling decision, (2) a total sessions counter, (3) an accepted sessions counter, and (4) the target sampling percentage. The counter pair drives the self-correcting algorithm. The session object enables session continuity — once a decision is made, every subsequent beacon for that session reads and returns the same status without recalculating.

### Q: Why is `samplingStatus` stored as a string rather than a JSON boolean?

Redis JSON module interop with Lua returns values as strings. Storing `"true"` and `"false"` as string values avoids type coercion issues when reading back from Redis via `JSON.GET` inside Lua and when parsing the return value back in Java via Jedis.
