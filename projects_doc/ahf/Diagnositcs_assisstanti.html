#1. Elevator Pitch

#30-Second Version

"The DCA module is a diagnostics intelligence layer inside the Oracle AI platform. It takes bug descriptions, support requests, attachment inventories, diagnostic file metadata, and user problem statements, then uses a mix of LLMs, embeddings, vector search, rule-based validation, and database orchestration to do four things: generate structured diagnostic knowledge from bugs, suggest the right tfactl diagcollect -srdc ... command, validate whether an existing collection is already sufficient, and classify on-demand diagnostic requests into file or command lookups."

#2-Minute Version

The DCA directory is not just one script. It is a suite of related workflows.

At the center, DiagGenerator mines bugs and diagnostics, asks an LLM to extract strict JSON outputs, validates file paths, predicts impact/symptom categories, and stores the resulting structured data in AHF/DDR databases. That output becomes reusable diagnostic knowledge.

Then there is DiagAdvisor, which takes a free-form problem description and turns it into a recommended diagnostic collection command. It first validates the description, may summarize it to fit token limits, extracts entities like error codes and keywords, semantically maps broad keywords to known SRDC keywords, performs embedding-based vector search against SRDC summaries, and if no strong SRDC match is found, falls back to an impact/symptom prediction pipeline.

Validator sits downstream of Advisor. It checks whether a support request already contains the right TFA collection or attachments. DiagChecker handles bug comments and ad-hoc requests, classifies requests into "find a file" vs "find a command", and checks if the requested evidence already exists in the collection.

So the overall value of DCA is: turn messy support/bug data into structured, searchable, automatable diagnostic guidance.

#5. DiagGenerator Deep Dive

DiagGenerator is the "knowledge creation" engine of the DCA suite. It processes bugs, extracts structured diagnostic intelligence, stores it in AHFDB, and can also build aggregate problem-type JSON objects.

#What it does

  • Fetches bugs by date range, problem type, explicit bug list, or default lookback window.
  • Gets bug descriptions from BugDB and status-30 comments from DDR-side data.
  • Predicts impact type and category by reusing ImpactSymptomPrediction.
  • Asks an LLM to extract strict keywords and raw diagnostic paths.
  • Gets AHF Web and DevShell file histories and asks an LLM to extract raw paths from them too.
  • Persists structured output into AHFDT_BUG_DIAGNOSTICS.
  • Optionally builds per-problem-type JSON by merging many bug records.
  • Supports scheduled execution with job tracker state and backlog/daily windows.
AI concept used here
In DiagGenerator, AI is used primarily for structured information extraction, not open-ended text generation. The prompts are deliberately strict and ask for JSON only. This reduces ambiguity and makes downstream persistence deterministic.

#Bug-to-Diagnostics Flow

1. Bug selection
   ├─ start_date/end_date → date range query
   ├─ problem_type → problem-type query
   └─ bugs list or default last-N-days

2. Per-bug summary processing
   ├─ Fetch bug subject/description from BugDB
   ├─ Append status-30 comments from DDR-side query if present
   └─ Predict category + impact type using ImpactSymptomPrediction

3. LLM strict extraction
   ├─ System prompt: bug_data_json_prompt.md
   ├─ Output JSON keys: keywords, raw_diagnostic_paths
   └─ Exact error/process/component tokens + raw file paths only

4. Diagnostic file mining
   ├─ Query AHF Web file history from TFA_WORKSPACE / TFA_WORKSPACE_ACTIONS
   ├─ Read DevShell history CLOBs in chunks using CHUNK_SIZE = 65536
   └─ Ask LLM to split results into ahf_web and devshell raw file paths

5. Persistence + aggregation
   ├─ Persist into AHFDT_BUG_DIAGNOSTICS
   ├─ Validate paths
   ├─ Merge bugs into ProblemJson
   └─ Update problem JSON store

#AI Concepts inside DiagGenerator

#1. Impact/Symptom Prediction Reuse

DiagGenerator inherits from ImpactSymptomPrediction and calls get_impact_type() to derive the likely category and impact type of a bug.

#2. Strict JSON Extraction

The LLM prompt is designed like an information-extraction contract: exact keywords only, exact raw paths only, no normalization.

#3. Structured Persistence

The AI output is never treated as final by itself. It is parsed using JsonUtils.get_json_object() and then stored in typed DB fields.

#4. Validation after AI

File paths extracted by the model are passed through DiagnosticPathValidator. This is a classic "AI + deterministic validator" pattern.

#Scheduler and Tracker Flow

The scheduler logic in DiagGenerator is more sophisticated than a simple cron job. It stores job state in AHFDT_JOB_TRACKER and computes either a backlog or daily window.

#6. DiagAdvisor Deep Dive

DiagAdvisor is the best file to discuss when someone asks: "Where is the AI in this system?" It is the module that most clearly demonstrates modern AI orchestration patterns.

#Core mission

Given a free-form problem description, generate the best possible tfactl diagcollect command. Prefer a specific SRDC command if possible; otherwise fall back to a generic command.

#Processing Flow

1. Input description
   └─ Free-form text from user, UI, API, or validator flow

2. Valid description?
   ├─ false → render error message
   └─ true  → continue

3. Description processing
   ├─ Use token estimate and safe token threshold
   ├─ Summarize only if needed for embedding safety
   └─ Extract entities: error codes, hostnames, db names, keywords, event time

4. Semantic normalization
   ├─ Load known keywords from SRDC metadata
   └─ LLM maps broad extracted keywords to canonical SRDC keywords

5. Embedding + hybrid retrieval
   ├─ Convert summary text to embedding using cohere.embed-english-v3.0
   ├─ Query AHFDT_SRDC_SUMMARY using similarity + keyword score + error score
   └─ Return top SRDC candidates

6. SRDC match found?
   ├─ Yes → return `tfactl diagcollect -srdc <name>`
   └─ No  → fallback to impact / symptom prediction

Figure 4 — DiagAdvisor: validation, summarization, normalization, embedding search, and fallback

#Why this is a strong AI design

StageAI TechniqueHow It Is Done in CodeWhy It Helps

Adaptive summarizationLLM summarization with token guardrailsgenerate_summary() only summarizes when estimated tokens exceed safe embedding limits.Preserves meaning while staying inside embedding limits.

#Important code behaviors

  • If the description is invalid, the workflow does not continue — it returns a user-friendly message.
  • If an SRDC is found via embeddings, the output is tfactl diagcollect -srdc <name>.
  • If not, the workflow maps predicted symptom → problem → SRDC using JSON resources such as symptoms_to_problem.json and problem_to_srdc.json.
  • If still nothing specific is found, it returns the safe default tfactl diagcollect.
  • Every execution writes audit metadata including latency, source API, similarity score, model, and output command.
Useful one-line summary
"DiagAdvisor is a production-style hybrid retrieval pipeline: validate → summarize if needed → extract entities → normalize vocabulary → retrieve by vectors and lexical signals → fallback to predictive classification."

#7. Validator & Attachments

Validator is a decision layer on top of DiagAdvisor. It answers the operational question: Do we already have the right collection in the SR/workspace, or should we ask the user to run one?

#How it works

  1. Validate that at minimum a workspace is present.
  2. Resolve the description using description_fetcher.py. It tries to read the latest matching XML under the workspace, extract <SRDescription_c>, and normalize whitespace.
  3. Call DSClient, which internally executes DiagAdvisor.
  4. If an SRDC is suggested, call AttachmentManager / AttachmentProcessor.
  5. Inspect attachment names; classify TFA zip collections vs non-TFA files.
  6. Check whether the recommended SRDC collection exists for all relevant hosts.
  7. Return a summary like "logs available" or "run collection".

   └─ latest XML → SRDescription_c
   │
   ▼

   └─ suggest collection command
   │

   ├─ scan attachment names
   ├─ detect .tfa*.zip
   └─ extract host + SRDC from zip naming

   └─ Yes → Return: logs available / collection already exists

Figure 5 — Validator converts an AI recommendation into an operational decision

#AI concept used here

Validator itself is mostly orchestration and evidence checking. The AI step is delegated to DiagAdvisor. This separation is good architecture: recommendation generation is AI-heavy, while verification of existing evidence is deterministic and rule-based.

#Attachment rules worth remembering

  • Files matching \.tfa.*\.zip$ are treated as TFA collections.
  • If the attachment name contains .tfa_srdc_<name>, the SRDC is extracted from the filename.
  • Collections are grouped by host and then compared against the recommended SRDC.
  • collection_exists becomes true only if the SRDC appears for all hosts in the grouping.

#8. DiagChecker

DiagChecker handles requests like: "For this bug, is the requested evidence already present in the collection?" It is a classification + lookup workflow.

#Flow summary

  1. Receive a bug monitoring payload or on-demand JSON payload.
  2. If on-demand, create a request tracker row in AHFDT_DC_REQUEST_TRACKER.
  3. Use an LLM prompt to categorize user intent into file lookup or command lookup.
  4. If file: check file presence in AHFDB.
  5. If command: embedding-search diagnostic summary docs and validate output files from the matched command.
  6. Update tracker rows with AI output, response comment, and run status.
ComponentResponsibilityTechnique
CategorizeQueryClassify request into file vs command.LLM JSON classification
DiagFileProcessorCheck if a file exists in collection and fetch absolute path.SQL lookup
DiagCommandProcessorSearch similar diagnostic commands using embeddings.Embedding search + file validation
AI concept used here
DiagChecker uses AI in two distinct ways: first for intent classification ("is the request about a file or a command?"), then for semantic retrieval when the request is command-like and must be matched to existing diagnostic summary embeddings.

#Advanced point

DiagCommandProcessor does not stop at semantic similarity. After it retrieves likely matching commands, it checks whether the command's output file is actually present in the bug collection. That means the system uses AI for candidate generation, and database evidence for final verification.

#13. Design Decisions & Tradeoffs

#1. Why strict JSON prompts?

Because downstream systems need machine-readable outputs. Free-form prose is bad for persistence, validation, and repeatable testing. So prompts explicitly say "JSON only" and define exact schemas.

#2. Why mix AI with validators?

Because AI can infer and normalize semantics, but it should not have the final say over production actions. This project correctly uses deterministic checks after AI output — for example path validation, file existence checks, SRDC existence in attachments, and DB-backed trackers.

#3. Why hybrid retrieval instead of only embeddings?

Because support descriptions often contain highly precise signals like ORA-00600, CRS-8011, or named events like CHA_LOG_SYNC_HANG. Pure vector search could miss the importance of those. Hybrid retrieval preserves lexical precision while still benefiting from semantic similarity.

#4. Why use fallback prediction?

Because no SRDC document collection is ever perfect. If semantic retrieval cannot confidently map a description to an SRDC, the system still has a second route: impact/symptom prediction → symptom-to-problem mapping → problem-to-SRDC mapping.

#5. Why store audit data?

Two reasons: observability and trust. In production AI systems, it is important to know what input produced what output, via which model, in how much time, and with what confidence-like scores.

#6. Why chunk DevShell CLOB reads?

Because DevShell records can be large CLOBs. Reading in fixed-size chunks avoids high memory spikes and makes DB driver interaction safer.

Core tradeoff
The system tries to maximize automation without losing determinism. That is the central architectural tradeoff across almost every module in this codebase.

#14.1 Prompting Strategies

#What is a prompt?

A prompt is the instruction you give an LLM. In DCA, prompts are loaded from Markdown files (prompts/ directory) via prompt_manager.py. This approach is good because prompts can be versioned, reviewed, and swapped without code changes.

#System prompt vs user prompt

  • System prompt: Sets the LLM's role, constraints, and output format. In DCA this is where "JSON only, no explanation" lives.
  • User prompt: The actual input — bug text, description, entity list, etc.

Using separate system and user roles is a best practice for structured extraction because it separates what the model is from what it is given.

#Zero-shot, few-shot, and chain-of-thought

TechniqueWhat It MeansDCA Usage
Zero-shotNo examples in the prompt; model must generalize from instructions alone.Keyword mapping and binary validation prompts.
Few-shotA small number of input-output examples are included in the prompt to guide the model.Path extraction prompts may include example bugs and their expected JSON outputs.
Chain-of-thought (CoT)Prompt instructs the model to reason step by step before answering.The reflection step in ImpactSymptomPrediction effectively does this — it reasons about candidates before producing final ranks.

#Prompt constraints used in DCA

  • Output in JSON only, no prose.
  • Do not normalize or infer; extract exactly as written.
  • Do not add keys not specified in the schema.
  • If a field is absent in input, return an empty array, not null.

These constraints are hallucination-mitigation measures baked into the prompt design.

#14.2 Tokenization and Context Window

#What is tokenization?

LLMs do not read raw text; they read tokens — sub-word units that balance vocabulary size and coverage. "ORA-00600" might tokenize as 4–5 tokens. Knowing this matters because LLMs have a maximum context window in tokens.

#Why context window matters in DCA

DiagAdvisor estimates token count of the input description before deciding whether to summarize. If the raw description is short enough, it is used directly. If it would exceed the embedding model's safe token threshold, it is first summarized by the LLM to a shorter version that preserves meaning.

This prevents:

  • Truncation mid-sentence (the embedding API just cuts off).
  • Loss of key signals like ORA codes appearing later in long text.

#Practical token math

A rough rule: 1 token ≈ 0.75 English words. So a 3,000-word SR description ≈ 4,000 tokens. If the embedding model's limit is 2,048, the description must be summarized first.

#14.3 Embeddings — Concept to Implementation

#What are embeddings?

An embedding is a fixed-length numeric vector that represents the semantic meaning of text. Texts that are semantically similar will have vectors that are close in high-dimensional space (measured by cosine similarity or dot product).

#How DCA uses embeddings

        ▼

#Why Cohere's embedding model?

cohere.embed-english-v3.0 is an enterprise-grade embedding model optimized for semantic search and retrieval. Oracle's AI platform provides access to it. It produces better-quality retrieval vectors than basic sentence transformer models.

At query time, the runtime embedding of the user's description is compared against every stored SRDC embedding.

#14.4 Vector Search and Similarity Scoring

#Cosine similarity

Given two vectors A and B:

cosine_similarity(A, B) = (A · B) / (||A|| × ||B||)

A score of 1.0 = identical meaning. A score close to 0 = unrelated. A score of -1.0 = opposites (rare in practice for document embeddings).

#Why not Euclidean distance?

Cosine similarity is magnitude-independent; it cares only about direction. This is more robust for document embeddings where one description might be 50 words and another 500 words.

#Hybrid scoring in DCA

final_score = w1 × vector_similarity
            + w2 × keyword_score
            + w3 × error_score
  • vector_similarity: from embedding cosine match.
  • keyword_score: how many canonical SRDC keywords appear in the description.
  • error_score: whether exact Oracle error codes (ORA-xxxxx, CRS-xxxxx) appear.

This is better than pure vector search because:

  • ORA-00600 in the description should strongly boost internalerror SRDC, not just nudge it.
  • Pure embedding might semantically associate "internal" with other SRDCs.

#Top-K retrieval

The query returns the top K most similar SRDC candidates. The system then applies a confidence threshold to decide if the best candidate is good enough to recommend, or whether the fallback path should be used.

#14.5 RAG — Retrieval Augmented Generation

#What is RAG?

RAG is an architecture where you:

  1. Retrieve relevant documents or data from a knowledge store.
  2. Augment the LLM's prompt with that retrieved content.
  3. Generate a response that is grounded in the retrieved data, not just the LLM's training weights.

#RAG vs pure LLM

AspectPure LLMRAG
Knowledge sourceTraining data only.External, up-to-date knowledge store.
Hallucination riskHigh for domain-specific facts.Lower, as model is grounded in retrieved text.
FreshnessStale (training cutoff).Current as the index.
CustomizationHard without fine-tuning.Easy by updating the knowledge store.

#How DCA implements RAG (and why it is better)

DiagAdvisor is RAG, but hardened:

  • The knowledge store (AIDB) is structured, versioned, and managed.
  • Retrieval uses hybrid scoring, not just vector similarity.
  • The LLM does ranking and reflection, not open-ended generation.
  • The final output is a controlled command, not free text.

This is sometimes called structured RAG or constrained RAG — the generation phase is heavily constrained to ensure the output is operationally safe.

#14.6 LLM Ranking and Reflection

#What is LLM-as-ranker?

Instead of using vector similarity scores alone to rank candidates, you pass the top candidates to the LLM and ask it to choose and rank them with reasoning. This works because the LLM can apply nuanced semantic judgment the scoring function cannot.

In ImpactSymptomPrediction:

  1. Top K impact/symptom candidates are retrieved by vector search.
  2. The LLM is asked: "Given this description and these candidates, which are most likely and in what order?"
  3. The LLM outputs a ranked list with confidence and brief reasoning.

#What is reflection?

Reflection is a second LLM call that reviews the first output and refines it. In this project:

  1. First pass: rank candidates, produce summary and root cause.
  2. Second pass (reflection): review the first output, check consistency, refine rankings, improve root cause explanation.

This is analogous to how a good engineer reviews their own code before submitting it. The cost is one extra LLM call per execution. The benefit is meaningfully better output quality.

#Reflection vs chain-of-thought

ConceptStructure
Chain-of-thoughtSingle pass where the model reasons before answering.
ReflectionTwo separate passes — the second pass explicitly critiques and refines the first.

Both improve quality; reflection is stronger for complex ranking tasks where initial output can be inconsistent.

#14.7 Token Limits, Summarization, and Chunking

#Why summarization happens before embedding

Embedding models have a maximum token limit (often 512 or 2048 tokens depending on the model). If you feed in a 6,000-token SR description, it will be truncated. Truncation is dangerous because critical signals like the ORA error code might appear at token 3000 and get cut off.

DiagAdvisor's flow:

1. Estimate token count of raw description
2. If count > safe_threshold → summarize with LLM first
3. Use the summary as the embedding input

The summarization prompt tells the model to preserve error codes, event names, and product identifiers — exactly the signals the retrieval step needs.

#Why chunking for CLOB reads

DevShell history is stored as CLOBs (Character Large Objects) in the database. A CLOB can be multiple megabytes. Reading the whole thing into memory at once is risky. DiagGenerator reads in chunks of CHUNK_SIZE = 65536 bytes (64 KB), which:

  • Keeps memory usage bounded.
  • Is safe across different JDBC and Python DB driver versions.
  • Allows incremental processing.

#14.8 Temperature, Determinism, and Why It Matters

#What is temperature?

Temperature is a parameter passed to the LLM API that controls output randomness.

TemperatureBehavior
0.0Fully deterministic; always picks highest-probability token.
0.2–0.5Low creativity; good for structured extraction tasks.
0.7–1.0Higher creativity; good for open-ended text generation.
> 1.0Very random; usually not useful for production systems.

#DCA's use of temperature

For extraction prompts (keywords, paths, entities), temperature is kept low. This maximizes consistency: running the same prompt twice should produce the same JSON. High temperature here would mean the same bug could produce different keyword sets on different runs, breaking reproducibility.

For reflection and ranking prompts, slightly higher temperature might be acceptable since the model is reasoning, not just extracting. But even there, the output is validated before use.

#14.10 Confidence and Thresholds

#Why confidence matters

A vector search always returns a top match — even if the description is completely unrelated to any SRDC. Without a confidence threshold, the system would always recommend something, even when it should say "I don't know."

DCA handles this by:

  1. Using the hybrid score as a proxy for confidence.
  2. Applying a minimum score threshold below which no SRDC is returned.
  3. Falling back to ImpactSymptomPrediction when no SRDC is above threshold.
  4. Falling back to generic tfactl diagcollect when even that cannot provide confident output.

This is a graceful degradation pattern — always return the most specific safe answer you can, and be explicit when you cannot be specific.

#Confidence calibration gap

One honest gap in the current design: LLM outputs do not provide calibrated probabilities. When the LLM says "confidence: high", that is a qualitative assessment, not a statistically calibrated probability. Future improvement could involve:

  • Comparing model confidence to actual correctness over historical audit data.
  • Using ensemble voting across multiple models for higher-stakes decisions.

#15. Hallucination Handling — Deep Dive

Interview Question: "How do you handle hallucinations here?"

This deserves a dedicated section because it is the most common and most important question interviewers ask about any production LLM system.

#What hallucination means in DCA's context

A hallucination in this project would be:

  • The LLM inventing a file path like /u01/oracle/fake_trace/nonexistent.trc that was never in the bug text.
  • The LLM inventing an SRDC name like dbfixall that does not exist in the known SRDC catalog.
  • The LLM extracting a keyword like LGWR_FIX which is a plausible-sounding but non-existent Oracle process name.
  • The model producing malformed JSON that breaks the persistence pipeline.

Any of these would cause real operational problems: wrong diagnostic guidance, broken DB inserts, or wasted support engineer time.

#DCA's multi-layer hallucination defense

DCA does not rely on a single defense. It uses at least five distinct layers:

#Layer 1: Strict prompt constraints

The prompts explicitly forbid the model from:

  • Inventing data not present in the source text.
  • Normalizing paths (e.g., converting /u01/ to /opt/).
  • Adding extra JSON keys not specified in the schema.
  • Returning prose explanations alongside JSON.

This reduces the surface area for hallucination by leaving the model less room to improvise.

#Layer 2: Closed-vocabulary keyword mapping

In KeywordMapper, when the LLM is asked to normalize extracted keywords to SRDC vocabulary, it is given the full list of known canonical keywords as part of the prompt. The model is instructed to only use keywords from that list, not invent new ones.

This is called a constrained decoding approach at the prompt level: the model cannot invent a keyword because the prompt tells it to select from a fixed set.

#Layer 3: Post-extraction path validation

After the LLM extracts raw diagnostic paths, DiagnosticPathValidator runs every path through:

  • Placeholder detection: {hostname}, {db_name}, etc. are flagged.
  • Glob detection: paths containing * or ? are flagged.
  • Template detection: paths with parenthesized fragments are flagged.

Invalid paths are logged and excluded from persistence. The system never stores a hallucinated path in AHFDT_BUG_DIAGNOSTICS unchecked.

#Layer 4: SRDC existence verification

DiagAdvisor retrieves SRDC candidates from AHFDB/AIDB — the only SRDCs that exist in those stores are real, validated SRDCs loaded by the platform team. The model never generates an SRDC name from scratch; it retrieves from a known set. This architectural choice eliminates an entire category of hallucination.

#Layer 5: JSON schema validation and fallback

All LLM outputs are parsed through JsonUtils.get_json_object(). If the model returns malformed JSON, non-JSON prose, or a response with missing required fields:

  • The parse fails gracefully.
  • An error is logged.
  • The workflow falls back to a safe default (empty result, generic command, or retry).

The system never crashes or produces corrupt DB state because of a bad model response.

#The overall philosophy

"In production AI systems, assume the model will occasionally be wrong. Build every layer of your pipeline to tolerate that, contain it, and recover from it gracefully."

That is exactly what DCA does. The model is trusted to handle semantic ambiguity and natural language understanding. Every other decision — path validity, SRDC existence, collection coverage — is handled by deterministic code or database lookups.

#What to say in an interview

"We handle hallucinations at multiple layers. First, prompts are strict: JSON only, no invention, extract only what's in the source. Second, for keyword normalization the model is given a closed vocabulary — it can only pick from known SRDC keywords. Third, all file paths extracted by the model go through a deterministic validator that filters placeholders, globs, and invalid templates. Fourth, SRDC selection is retrieval-based from a curated store, so the model cannot invent an SRDC name. Fifth, all LLM outputs are parsed and schema-checked; malformed responses trigger a safe fallback, not a crash."

#16. Scale: Processing Millions of Bugs in DiagGenerator

Interview Question: "Suppose you have lakhs of bugs — how does DiagGenerator process those for regex generation?"

This question tests whether you understand the operational mechanics of DiagGenerator beyond just the happy path.

#The core challenge

If DiagGenerator had to process 100,000+ bugs (a realistic historical corpus) in a single run:

  • LLM calls are expensive: each bug requires at least one call for keyword/path extraction and one for AHF Web/DevShell mining.
  • DB queries need to be efficient: fetching 100,000 bug descriptions row by row is slow.
  • Memory must be managed: you cannot hold the entire bug corpus in RAM.
  • The system must be resumable: if it fails at bug #50,000, restarting from scratch wastes resources.

#How DiagGenerator handles this

#1. Date-range windowed chunking

DiagGenerator does not process all bugs in one shot. Even in backlog mode, it splits the historical window into chunks of at most MAX_CHUNK_DAYS = 30 days. So a 365-day backlog becomes roughly 12 sequential chunks of 30 days each.

backlog window: 365 days
       │
       ▼
chunk 1: days 1–30
chunk 2: days 31–60
...
chunk 12: days 331–365

Within each chunk, only the bugs modified or created in that window are fetched. This keeps query result sizes manageable.

#2. Job tracker checkpointing

AHFDT_JOB_TRACKER stores the last successful run date. If the system crashes mid-processing, the next run resumes from the last checkpoint rather than restarting at day 1. This is critical for processing large historical backlogs reliably.

Run 1: processes days 1–30. Saves checkpoint: "last_success = day 30".
Run 2: resumes from day 31.
...

#3. Per-bug LLM call batching and error isolation

Each bug is processed independently. If the LLM call for one bug fails (timeout, rate limit, bad response), only that bug is skipped — the loop continues with the next one. This is error isolation at the unit level.

In practice this means the system never fails the entire batch because of one bad bug or one transient LLM API error.

#4. CLOB reading in chunks

For DevShell history, which can be megabytes per bug, the code reads the CLOB in CHUNK_SIZE = 65536 byte chunks and accumulates text progressively. This prevents out-of-memory errors when processing bugs with large diagnostic histories.

#5. Persistence strategy: insert-then-update

For each bug, persist_json_in_ahfdb() tries an INSERT first. If a row already exists (integrity constraint), it falls back to UPDATE. This means re-running DiagGenerator over the same window is idempotent — it updates existing records rather than duplicating them.

#6. Problem JSON aggregation is a separate pass

create_problem_json() runs as a distinct, aggregation-only step after individual bug records are persisted. It reads from AHFDT_BUG_DIAGNOSTICS and merges keyword/diagnostic frequencies across all bugs in a problem type. Because it reads already-persisted data rather than re-calling the LLM, it is fast and can handle any number of bugs.

#For regex generation specifically

Regex metadata generation (generate_regex_metadata.py) works on already-stored diagnostic paths, not raw bugs. By the time regex metadata is generated:

  1. All bugs have been processed by the LLM.
  2. Paths have been validated and stored in AHFDT_BUG_DIAGNOSTICS.
  3. generate_regex_metadata.py reads those stored paths and tokenizes them for Lucene indexing.

This means regex generation is a post-processing step over structured DB data, not a real-time LLM-driven step. It can be run as a batch job over the full history of stored diagnostics without any LLM involvement, making it fast and scalable.

#What to say in an interview

"DiagGenerator handles scale through several mechanisms. First, it never processes all bugs at once — it uses date-range chunking of up to 30 days per window, so even a year's backlog becomes 12 manageable chunks. Second, job tracker checkpointing means a crash doesn't require starting from scratch. Third, each bug is processed independently, so one LLM failure doesn't abort the batch. Fourth, CLOB data is read in fixed-size chunks to avoid memory spikes. Fifth, regex metadata generation is a separate post-processing step over already-stored structured data — it doesn't involve LLM calls at all, so it scales cheaply."

#17. DiagGenerator Run Frequency & Scheduling

Interview Question: "How often do we run DiagGenerator?"

#Two modes, two frequencies

DiagGenerator has two distinct operational modes controlled by the schedular flag and the job tracker state.

#Daily mode (steady state)

Once the historical backlog is processed, DiagGenerator runs in daily mode. It processes only bugs from the last rolling day-sized window since the last successful checkpoint.

  • Trigger: Scheduled job (cron or Oracle AHF job scheduler).
  • Frequency: Once per day, typically during off-peak hours.
  • Window: ~1 day of new/updated bugs.
  • Volume: In most environments, this means tens to a few hundred bugs per day.
  • LLM cost: Low — only new bugs need processing.

#Backlog mode (initial bootstrap or recovery)

If no recent checkpoint exists (first run ever, or long gap due to failure), DiagGenerator switches to backlog mode.

  • Trigger: Absence of a recent last_success_run_date in AHFDT_JOB_TRACKER.
  • Lookback: Up to 365 days (HISTORICAL_LOOKBACK_DAYS = 365).
  • Chunking: Broken into 30-day sub-windows (MAX_CHUNK_DAYS = 30).
  • Resumable: Each chunk updates the checkpoint, so recovery mid-backlog is safe.

#What triggers a run in practice?

In Oracle's AHF platform, DiagGenerator is typically triggered by a platform-level job scheduler. The schedular=true flag is passed when the job is invoked this way. If invoked manually (e.g., for testing or backfill), schedular=false can be passed along with explicit start_date, end_date, or a bugs list.

#Why daily frequency makes sense

  • Bug data accumulates incrementally; daily processing keeps the knowledge base fresh.
  • LLM API costs are proportional to volume; processing only new bugs daily is cost-efficient.
  • The reflection and impact prediction steps involve multiple LLM calls per bug; batching them nightly avoids latency impact on interactive workflows like DiagAdvisor.

#What to say in an interview

"DiagGenerator runs in two modes. In steady state, it runs once daily, processing only bugs from the last day's window as tracked by a job tracker checkpoint in AHFDT_JOB_TRACKER. On initial bootstrap or after a long gap, it enters backlog mode, which processes up to 365 days of history in 30-day chunks. Each chunk updates the checkpoint, so the system is fully resumable if a run fails partway through."

#18. Knowledge Check Questions (27)

Below are practice questions arranged from straightforward to advanced. The answers are intentionally compact enough to revise quickly, but detailed enough to use in deep technical discussions.

1. What is the DCA project in one sentence?

DCA is a diagnostics intelligence suite that transforms bug data, support request context, and free-form problem descriptions into structured diagnostic knowledge, recommended TFA collection commands, validation decisions, and collection coverage checks.

2. What are the main workflows in this directory?

The four main workflows are DiagGenerator, DiagAdvisor, Validator, and DiagChecker. Generator creates structured knowledge, Advisor suggests commands, Validator checks whether existing collections are enough, and Checker verifies requested files or commands against bug collections.

3. Where exactly is AI used in this codebase?

AI is used in strict JSON extraction, description validation, summarization, entity extraction, keyword normalization, embedding generation, hybrid retrieval, impact/symptom ranking, and reflection. But final operational decisions are often validated via deterministic DB checks and regex/path logic.

4. How does DiagGenerator use an LLM?

It sends bug description text and comments to a strict prompt that extracts keywords and raw diagnostic file paths as JSON. It also sends AHF Web / DevShell diagnostic text to a second strict prompt that extracts raw file paths into ahf_web and devshell arrays.

5. Why is the word "strict" repeated in prompts?

Because the project needs reliable machine-readable output. The prompts explicitly forbid normalization, inference, extra explanation, or unstructured prose. That reduces hallucination risk and simplifies DB persistence.

6. How does DiagAdvisor choose an SRDC command?

It validates the description, summarizes it if token limits require, extracts entities, maps broad keywords to canonical SRDC keywords, embeds the summary, performs hybrid retrieval over stored SRDC summaries, and returns a tfactl diagcollect -srdc ... command if a strong candidate exists. Otherwise it falls back to impact/symptom prediction.

7. What is hybrid retrieval in this project?

Hybrid retrieval means the system does not rely only on embedding similarity. It combines vector similarity with lexical evidence such as Oracle error codes and canonical keywords. In the SQL, this appears as similarity, keyword_score, and error_score.

8. Why not use pure vector search only?

Because support diagnostics contain precise symbolic evidence like ORA codes, CRS codes, and specific event names. Those signals should strongly influence ranking. Hybrid retrieval keeps that precision while retaining semantic flexibility.

9. What does ImpactSymptomPrediction do?

It predicts top category/impact pairs for a bug or description. It summarizes text, generates embeddings, retrieves nearby impact/symptom vectors, asks an LLM to rank top candidates, and then performs a reflection pass to refine summary, root cause, and rankings.

10. What is the reflection step and why is it useful?

After initial ranking, the project makes a second LLM call to review the first output, identify the likely root cause, refine the summary, and improve rank ordering. This improves coherence and acts like a self-review stage.

11. How does Validator differ from DiagAdvisor?

DiagAdvisor answers "what collection should I run?" Validator answers "do I already have enough logs or collections in this workspace?" It uses Advisor for recommendation, then deterministically checks existing attachments and collection names.

12. How does Validator find the SR description?

It looks in the workspace directory for the latest matching XML file, using an epoch extracted from the filename. Then it extracts the content inside <SRDescription_c>. If that fails, it falls back to the description provided in the payload.

13. What does AttachmentProcessor check?

It inspects attachment names, separates TFA collections from non-TFA files, extracts SRDC names from TFA zip naming patterns, groups them by host, and decides whether the suggested SRDC collection already exists across all hosts.

14. What is DiagChecker's role?

DiagChecker handles bug-based or on-demand requests asking whether certain evidence already exists. It classifies the request into file or command mode, then either checks file presence directly or uses embedding search to match command-like intent to known diagnostic outputs.

15. How does DiagChecker use AI?

First, an LLM categorizes user intent as file-oriented or command-oriented. Second, for command-oriented requests, embeddings are used to retrieve likely diagnostic commands from stored summary docs. Final verification still depends on DB-backed file presence checks.

16. Why is QueryMapper a good design?

It centralizes SQL registration and lookup by DB category, keeps workflow code cleaner, and supports cached query retrieval. This reduces duplication and makes query ownership explicit.

17. How does the scheduler in DiagGenerator work?

When schedular=true, it loads the job tracker row, derives a processing window, and chooses between backlog mode and daily mode. It writes RUNNING status before processing, and updates SUCCESS or FAILED with checkpoint metadata after execution.

18. What is the difference between backlog mode and daily mode?

Backlog mode is used when no recent successful checkpoint exists. It starts as far back as 365 days and processes in chunks up to 30 days. Daily mode is used when the tracker is up to date and processes a smaller rolling day-sized window.

19. Why validate diagnostic paths after LLM extraction?

Because path extraction is safety-sensitive. The model may output placeholders, globs, or invalid templates. DiagnosticPathValidator filters and records invalid patterns, which is safer than trusting the model blindly.

20. What is `ProblemJson`?

It is an aggregation object that merges keywords and diagnostics across many bugs belonging to a problem type. The result is a reusable problem-level JSON containing keyword frequencies and diagnostic file frequencies.

21. Why does DescriptionProcessor summarize only when needed?

Because summarization can lose detail. The code estimates token usage and only summarizes when the text would exceed a safe embedding token threshold. That preserves fidelity for shorter descriptions while still protecting the embedding stage.

22. Why store SRDC summaries and embeddings in AIDB?

Because SRDC recommendation is fundamentally a retrieval problem. Storing descriptions, keywords, errors, and embeddings in a searchable store enables fast semantic matching across many known collection types.

23. What makes this project production-grade rather than a demo?

Several things: audit persistence, DB-backed state tracking, deterministic fallback logic, strict JSON extraction, prompt modularity, validators after AI output, hybrid retrieval instead of naive RAG, and request trackers for checker flows.

24. If you had to improve this project, what would you do next?

I would add stronger confidence calibration and routing thresholds, expand document search for attachment coverage, improve observability dashboards around audit data, add more typed schemas and validation around model outputs, and unify some repeated embedding / retrieval patterns across modules.

25. How do you handle hallucinations in this system?

At five layers: (1) strict prompts forbid invention and require exact extraction only; (2) keyword normalization uses a closed known-vocabulary list; (3) extracted paths go through DiagnosticPathValidator which filters placeholders, globs, and templates; (4) SRDC recommendation is retrieval-based from a curated store, not free generation; (5) all LLM outputs are JSON-parsed and schema-validated — malformed responses trigger safe fallback, not a crash.

26. How does DiagGenerator scale to process lakhs of bugs?

Through date-range chunking (max 30 days per window), job tracker checkpointing for resumability, per-bug error isolation so one failure doesn't abort the batch, CLOB reading in 64KB chunks, idempotent insert-then-update persistence, and a separate aggregation pass for ProblemJson that reads from already-stored data without additional LLM calls.

27. How often does DiagGenerator run and what decides the window?

In steady state it runs once daily, processing only the last day's new/updated bugs based on the job tracker checkpoint. On first run or after a failure gap, it enters backlog mode, processing up to 365 days of history in 30-day chunks, with each chunk saved as a checkpoint for recovery.
